# TodoSoftBei：软件杯赛题适配整改清单

## 目标

围绕第十五届中国软件杯赛题《基于多模态大模型技术的设备检修知识检索与作业系统》，将当前“工业故障诊断原型”重构为“设备检修知识与作业助手”，并按比赛评分维度推进功能、展示、证据和材料补齐。

## 赛题依据

- 赛题名称：`A基于多模态大模型技术的设备检修知识检索与作业系统`
- 组别：`A组`
- 出题企业：`龙芯中科技术股份有限公司`
- 赛题页：`https://www.cnsoftbei.com/content-15-1246-1.html`
- 本轮排除项：`LoongArch + 银河麒麟高级服务器版` 适配暂不纳入当前整改主线，仅在风险中保留说明。

## 当前结论

- 当前仓库已经具备 FastAPI、PostgreSQL、SSE、多智能体诊断和真实部署能力，但主体仍是“工业故障诊断原型”。
- 当前项目与赛题仍存在明显定义偏差：赛题主体是“检修知识检索 + 标准化作业 + 知识沉淀更新”，现有系统只覆盖了“智能分析子模块”的一部分。
- 当前优先级不是继续打磨旧诊断页，而是先冻结作品定义、知识库边界和演示主线，再开展赛题功能建设。

## 执行规则

- 执行顺序严格按 `TODO-SB-1` 到 `TODO-SB-8` 推进。
- 当前赛题整改以本文件为唯一任务源；历史 MVP/部署整改仍参考 `todo1.md`。
- 每完成一个主任务，必须同步更新 `progress.md`。
- 关键赛题决策、作品边界和技术取舍必须同步更新 `findings.md`。
- 当前诊断接口保留，但只能作为“智能分析子模块”，不得继续作为主产品定义。

### TODO-SB-2 检修知识库与知识检索主体建设

- 问题描述：当前只有传感器数据分析，没有检修手册、案例、标准步骤组成的知识库。
- 对应赛题要求：多模态知识检索、快速调取检修手册等资源。
- 当前项目现状：无知识文档入库、无检索索引、无出处引用。
- 影响范围：后端模型、数据库 schema、搜索接口、前端检索结果区。
- 目标状态：建立最小可用知识库，并支持文本/型号驱动的知识检索。
- 实施步骤：
  1. 定义最小知识实体：知识文档、知识分段、设备型号、故障类型、检修案例、知识关系。
  2. 先接入三类知识源：官方/提供手册、整理案例、标准步骤。
  3. 设计文档导入流程：上传 -> 切片 -> 元数据保存 -> 全文索引 -> 关系建立。
  4. 实现检索接口，输出标题、命中片段、来源、章节/页码和推荐理由。
  5. 明确诊断模块只做补充分析，不替代检索主体。
- 交付物：
  - 知识数据模型
  - 文档入库流程
  - 知识检索 API
  - 检索结果展示页
- 验收标准：
  - 输入文本或设备型号后，能返回带出处的检修知识结果。
- 依赖关系：依赖 `TODO-SB-1`。
- 状态：已完成（最小后端闭环）

### TODO-SB-3 多模态输入补齐

- 对应赛题要求：多模态知识检索。
- 当前项目现状：没有图片上传入口、没有设备型号检索入口、没有跨模态融合逻辑。
- 影响范围：前端输入区、知识检索接口、媒体文件处理、结果展示。
- 目标状态：支持文本、设备型号、单张故障图片联合驱动检索。
- 实施步骤：

  2. 新增图片上传接口与前端入口。
  3. 将图片先转成故障描述/部件标签，再复用文本检索链路。
  4. 统一检索返回结构，区分用户输入、系统识别结果、命中知识。
  5. 至少做通一条可稳定演示的图片场景。
- 交付物：
  - 多模态检索接口
  - 图片上传与识别流程
  - 多模态检索页面
- 验收标准：
  - 文本、型号可独立检索。
  - 至少一条“图片 -> 识别 -> 检索 -> 引用”链路稳定可演示。
- 依赖关系：依赖 `TODO-SB-2`。
- 状态：已完成（最小后端 + 静态入口页）

### TODO-SB-4 标准化作业指引闭环

### TODO-SB-5 知识沉淀、审核与人工修正

- 问题描述：当前没有案例上传、审核、知识回流和人工纠错能力。
- 对应赛题要求：支持案例上传，经审核后纳入知识图谱；支持手动标注与修正大模型输出。
- 当前项目现状：系统只消费数据，不支持一线经验沉淀。
- 影响范围：数据模型、后台流程、前端案例页、模型反馈闭环。
- 目标状态：形成“案例上传 -> 审核 -> 入库 -> 后续命中”的回流链路。
- 实施步骤：
  1. 设计案例上传表单和附件字段。
  2. 建立两态审核流：待审核、已通过/已驳回。
  3. 增加人工修正入口，允许对检索结果与模型输出补充说明。
  4. 将审核通过案例纳入知识库关系表。
  5. 保留修正记录，支撑测试报告和答辩展示。
- 交付物：
  - 案例上传接口
  - 审核接口
  - 人工修正能力
- 验收标准：
  - 新增案例可在审核后被后续检索命中。
- 依赖关系：依赖 `TODO-SB-2`。
- 状态：已完成（最小后端 + 静态案例页）
