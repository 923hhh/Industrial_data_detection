# 软件杯项目技术评审结论：实现状态分层

## 1. 文档目的

本文用于基于外部技术建议与当前仓库真实实现情况，给出一份更贴近工程事实的正式评审结论。重点不是罗列技术名词，而是判断：

- 哪些能力已经落地
- 哪些能力已有基础但尚未做深
- 哪些能力当前仍未实现
- 哪些方向现在不适合优先投入

本文默认场景为软件杯比赛与演示交付，不以商用工业平台的完整成熟度作为判断标准。

## 2. 评审对象

本次评审主要围绕四类外部建议展开：

1. 大模型异构部署与全栈开发技术
2. 多模态检索增强生成技术
3. 智能体工作流与规则引擎技术
4. 知识图谱与模型对齐微调技术

### 2.1 比赛环境约束补充

在第十五届中国软件杯 A 组赛题背景下，还必须额外纳入以下硬约束：

- 目标运行环境需兼容 `LoongArch + 银河麒麟高级服务器操作系统 V11 / V10`
- 目标测试硬件仅按 `4 核 CPU + 8GB 内存 + 256GB 硬盘` 估算
- 不默认存在高性能 GPU，因此本地 7B/8B 模型部署不应成为主链路前提
- 稳定的 `云端大模型 API + 流式通信` 能力，重要性高于“本地大模型是否完整落地”
- `Milvus / Neo4j` 等重型向量库或图数据库不适合作为当前比赛交付的必备技术栈
- 比赛交付标准不仅包含代码和系统，还包含需求、设计、测试、部署、PPT 和视频材料

## 3. 总体结论

外部建议中的很多方向本身没有问题，但把“长期演进路线”和“当前项目硬缺口”混在了一起。

对当前仓库更准确的判断应该是：

- `前后端分离、流式通信、Agent 编排、工具调用、Guardrails、OCR 导入、检索重排、知识定位、评测指标` 已经不是空白，部分能力已经完成正式工程化
- `本地模型异构部署、图文统一 embedding、多模态向量检索、严格 FSM 门禁、知识 Patch 化版本演进` 属于已有方向但尚未做深
- `GraphRAG / Neo4j、RLHF / DPO、LoRA / QLoRA 训练飞轮、重型向量基础设施` 现阶段不应被视为主线硬缺口

在这些约束下，简化判断应修正为：

`当前最该补的是轻量化工程闭环与比赛交付完整度，不是先堆重型平台。`

## 4. 已实现

以下能力已经在仓库中具备正式实现，不应再被简单归类为“还没有做”。

### 4.1 前后端分离与流式通信

- 后端已使用 `FastAPI` 构建正式 API
- 正式前端已使用 `Next.js + React + TypeScript`
- 已有 `SSE` 流式接口，不仅存在于诊断页，也已接入正式 `/agents` 主链路
- 前端已通过 `EventSource` 展示 Agent 阶段性事件与执行进度

结论：

`前后端分离 + 流式通信机制` 已实现。

### 4.2 Agent 编排基础能力

- 项目已有 `LangGraph` 多智能体背景与工作流基础
- 正式业务链中已经有：
  - Agent run 持久化
  - 工具调用链
  - 人工授权标记
  - 拦截项输出
  - 执行回放
- `/agents` 不再只是静态结果页，而是正式业务协作页

结论：

`Agent 编排框架基础` 已实现。

### 4.3 规则校验与安全守卫基础

- 当前系统已具备 Guardrails 层
- 已有：
  - `blocking_issues`
  - `authorization_required`
  - 安全前置条件校验
  - 工具执行风险状态
- 已能在高风险场景下要求人工授权，不是完全自由生成

结论：

`规则引擎基础版 / 安全守卫基础版` 已实现。

### 4.4 OCR、PDF 解析与知识导入工程链路

- 已支持正式图片上传入口
- 已支持 OCR 预览与导入
- 已支持 PDF 预览与导入
- 已支持知识分段、来源回溯、导入记录、文档筛选
- 导入已异步任务化，不再完全阻塞请求

结论：

`文档解析与预处理链路` 已实现。

### 4.5 检索增强与知识可定位能力

- 已有 PostgreSQL 全文检索优化
- 已有 rerank
- 已有 `retrieval_score / rerank_score`
- 已有知识锚点：
  - `section_path`
  - `step_anchor`
  - `image_anchor`
- 已支持从任务、案例、知识结果回看来源文档与命中位置

结论：

`检索增强基础版 + 来源定位能力` 已实现。

### 4.6 工程观测与评测闭环

- 已有请求 ID、统一错误码、基础指标日志
- 已有知识导入异步任务状态与恢复逻辑
- 已有固定评测脚本与业务指标汇总
- 工作台首页已能展示评测快照与运行指标

结论：

`基础可观测性 + 评测闭环` 已实现。

## 5. 部分实现

以下方向并非空白，但当前实现还不足以支撑“完全实现”这一判断。

### 5.1 大模型异构部署

当前情况：

- 仓库已具备多模型接入和 provider 级别切换基础
- 但还没有形成正式的统一模型网关
- 还缺：
  - 云端优先 / 本地轻量兜底的明确路由策略
  - 本地模型与云端模型统一路由
  - 超时降级
  - 熔断
  - 成本与时延统计
  - 面向本地算力的推理实例管理

评审结论：

`部分实现`

更准确地说，当前是“模型调用工程基础已具备，但异构部署平台能力未完成”。
在比赛约束下，更关键的缺口也不是“先把本地 7B 模型跑起来”，而是“先把云端模型调用做稳、做可切换、做可降级”。

### 5.2 多模态检索增强生成

当前情况：

- 已支持文本检索、图片上传、OCR 文本化、图片线索辅助
- 已支持结合设备型号、故障类型和知识重排
- 但还没有真正进入“图文统一 embedding + 统一向量空间召回”阶段

评审结论：

`部分实现`

当前更接近“多模态输入增强的检索系统”，还不是严格意义上的 `Multimodal RAG` 完整形态。

### 5.3 FSM 与强制证据门禁

当前情况：

- 已有 Guardrails、阻断项和人工授权
- 已有规则化校验思路
- 但还没有形成严格的步骤状态机，例如：
  - 某一步必须验证特定图片
  - 某一步必须满足特定遥测阈值
  - 某一步必须完成人工签核才能进入下一步

评审结论：

`部分实现`

当前是“规则层已建立”，但还不是“硬状态机驱动的工业流程控制”。

### 5.4 知识沉淀与更新

当前情况：

- 已支持案例审核后回流入库
- 已支持人工修正与审核
- 已形成知识再沉淀链路
- 但还没有做成：
  - Patch 化审核
  - 版本化知识管理
  - 原文与修改建议的结构化 diff

评审结论：

`部分实现`

当前更接近“审核入库”，尚未升级到“知识版本演进系统”。

## 6. 尚未实现

以下能力若严格按外部建议理解，当前仓库确实还没有正式落地。

### 6.1 真正的本地异构大模型部署栈

当前尚未正式落地：

- `vLLM`
- `TensorRT-LLM`
- `Llama.cpp`
- `GPTQ / AWQ / INT8 / INT4` 量化部署链
- 本地推理性能基准和调度策略

评审结论：

`尚未实现`

补充判断：

`尚未实现` 不等于 `当前必须优先补齐`。在 `4 核 CPU + 8GB 内存` 约束下，这一项更适合作为加分项，而不是主验收标准。

### 6.2 真正的图文统一 embedding 与专业向量数据库

当前尚未正式落地：

- `CLIP / LLaVA / Qwen-VL` 等图文统一 embedding 召回链
- `Milvus / Qdrant / FAISS` 作为正式知识检索底座
- 图片、文本、OCR、设备标签进入同一向量空间后的统一近邻搜索

评审结论：

`尚未实现`

### 6.3 严格工业 FSM 门控执行

当前尚未正式落地：

- 步骤级状态迁移图
- 必须满足前置证据后才可推进的硬门禁
- 每类设备、每种检修等级对应的正式多分支状态机模板

评审结论：

`尚未实现`

### 6.4 Patch 化知识审查与版本关系图

当前尚未正式落地：

- 原知识片段与新建议之间的结构化 patch
- patch 审核记录
- 文档版本树或知识演进关系链

评审结论：

`尚未实现`

## 7. 不建议当前做

以下方向有长期价值，但不适合在当前比赛/MVP 阶段作为主线投入。

### 7.1 不建议当前先上 Neo4j / GraphRAG

原因：

- 当前真正缺的是 Patch 化审核、版本化管理和证据链
- 在关系建模尚未稳定时，先引入图数据库收益偏低
- 会显著增加建模、部署、查询和答辩解释成本

评审结论：

`不建议当前做`

### 7.2 不建议当前先做 RLHF / DPO / LoRA / QLoRA

原因：

- 目前尚未沉淀足够规模、足够干净的人工修正样本
- 现在的主要问题仍在业务流程结构和知识演进结构，而不是先训练模型
- 过早引入训练/MLOps 管线，会让项目复杂度陡增

评审结论：

`不建议当前做`

### 7.3 不建议当前先重构为重型向量平台

原因：

- 当前知识规模和比赛场景下的查询压力，未必需要优先替换检索底座
- 现阶段更值钱的是“统一 embedding 路径”和“可解释的命中定位”
- 即使未来要上向量库，也应放在检索语义路径成熟之后
- 在 `4 核 CPU + 8GB 内存` 环境下，重型检索平台还会额外放大内存和部署压力

评审结论：

`不建议当前做`

### 7.4 不建议当前以“自动控制设备”为目标继续扩展

原因：

- 当前系统更适合定位为“知识检索 + 作业建议 + 风险校验 + 人工授权”
- 直接迈向自动控制会显著抬高安全风险和评审质疑点

评审结论：

`不建议当前做`

### 7.5 不建议当前把“本地 7B/8B 大模型完整部署”作为主线目标

原因：

- 当前目标硬件资源过于受限
- 无 GPU 场景下，本地大模型极易拖垮响应时间和内存占用
- 比赛真正更看重的是主链路可运行、可演示、可解释，而不是是否把大模型塞进本地机器

评审结论：

`不建议当前做`

## 8. 最终评审意见

如果把外部建议翻译成更符合当前仓库的结论，可以归纳为：

### 已实现

- 前后端分离与 SSE 流式通信
- Agent 编排基础、工具调用、授权与拦截
- OCR / PDF 知识导入
- 检索重排、知识锚点与来源回看
- 基础可观测性与评测闭环
- 轻量 PostgreSQL 检索主栈与工程闭环

### 部分实现

- 大模型异构部署
- 多模态检索增强生成
- FSM 与强制证据门禁
- 知识沉淀后的版本化演进
- 国产化运行环境适配说明与统一模型网关

### 尚未实现

- 本地推理引擎与量化部署栈
- 图文统一 embedding 与正式向量数据库
- 严格工业状态机模板
- Patch 化知识审查与版本关系管理

### 不建议当前做

- Neo4j / GraphRAG 先行
- RLHF / DPO / LoRA / QLoRA 先行
- 重型向量基础设施先行
- 本地 7B/8B 模型完整部署先行
- 自动控制闭环先行

## 9. 建议的下一步

最合理的后续顺序建议为：

1. 先做 `统一模型网关 + 云端优先 / 本地轻量兜底 + 降级策略`
2. 再做 `严格 FSM + 步骤级证据门禁`
3. 然后做 `轻量多模态 RAG`，继续坚持 `PostgreSQL + OCR + rerank + 锚点定位`
4. 再做 `案例审核 Patch 化 + 知识版本化`
5. 最后才考虑 `图文统一 embedding、向量库升级、GraphRAG、偏好优化微调`

这一路线更符合当前仓库的真实基础，也更适合软件杯阶段的投入产出比。

## 10. 需补充到 PRD 与技术规划的内容

结合本次评审，以下内容必须明确写入正式需求与技术规划，而不能只停留在口头讨论中：

### 10.1 国产化与轻量化约束

- 运行环境必须考虑 `LoongArch + 银河麒麟`
- 主流程默认硬件按 `4 核 CPU + 8GB 内存 + 256GB 硬盘`
- 云端模型优先，本地模型仅作加分项
- 检索主栈继续以 `PostgreSQL + 全文检索 + 轻量 rerank` 为核心

### 10.2 关键 NFR

- 界面设计合理、美观、便于快速理解
- RAG 结果必须具备知识锚点和来源回看能力
- 后端必须具备统一错误码、Request ID 和基础指标日志

### 10.3 比赛交付标准

- 交付物不仅是系统代码
- 还必须包含需求文档、设计文档、产品说明书、测试报告、安装包与部署文档
- 初赛还需额外准备源码包、功能演示 PPT 与 7 分钟以内功能演示视频
