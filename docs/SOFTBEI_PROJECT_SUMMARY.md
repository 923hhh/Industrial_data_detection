# 项目总结：设备检修知识与作业助手

最后更新：`2026-03-31`

## 1. 项目一句话说明

本项目当前定位为面向软件杯赛题《基于多模态大模型技术的设备检修知识检索与作业系统》的参赛作品，作品名称已冻结为：

`设备检修知识与作业助手`

它的核心目标不是再做一个“工业故障诊断原型”，而是围绕设备检修场景，形成一条完整业务主线：

`知识检索 -> 作业指引 -> 步骤执行 -> 案例沉淀 -> 审核回流`

当前演示对象已冻结为：

`摩托车发动机检修`

原有工业传感器诊断与 SSE 流式分析能力仍保留，但现在只作为：

`智能分析子模块`

## 2. 这个项目现在已经实现了什么

### 2.1 知识库与知识检索

已实现能力：

- 支持检修知识文档入库，形成 `knowledge_documents` 与 `knowledge_chunks` 两层知识结构。
- 支持按文本、设备类型、设备型号、故障图片进行多模态检索。
- 支持返回命中摘录、来源文件、章节/页码、推荐理由。
- 支持长中文故障描述的 query rewrite、同义词扩展和“具体型号命中通用手册”策略。
- 支持正式知识中心中的文档列表、文档筛选、来源回溯、分段预览。

业务作用：

- 让一线检修人员不再只靠经验或整本翻手册，而是能快速定位到当前故障相关的知识条目。
- 让系统输出带出处的结果，便于复核、答辩展示和后续标准化使用。

### 2.2 OCR 与知识导入管理

已实现能力：

- 支持 PDF 手册导入。
- 支持图片型知识导入。
- 支持导入前预览、导入后任务记录、失败原因查看。
- 支持 OCR/视觉抽取后的处理提示展示。
- 正式前端已具备图片上传检索入口和图片型知识导入入口。

业务作用：

- 降低知识资产入库门槛，让手册、截图、图片型资料都能进入统一知识库。
- 让评委看到项目不只是“搜索框 + 模型回答”，而是具备知识资产管理能力。

### 2.3 标准化检修任务闭环

已实现能力：

- 可根据知识引用生成检修任务。
- 可按模板自动生成标准化步骤。
- 支持步骤执行状态更新、历史查看和结果导出。
- 步骤中已包含风险提示、注意事项和确认动作。

业务作用：

- 把“检索到知识”进一步转化为“可执行的作业流程”。
- 让系统从知识查询工具升级为检修作业辅助系统。

### 2.4 案例沉淀、审核与知识回流

已实现能力：

- 支持上传检修案例。
- 支持人工修正检索结果、模型输出和总结内容。
- 支持案例审核。
- 审核通过后自动回流为知识文档，参与后续检索。

业务作用：

- 形成“经验沉淀 -> 审核 -> 回流知识库 -> 下次复用”的闭环。
- 体现项目不只是消费知识，也能持续积累一线经验。

### 2.5 多智能体协作主线

已实现能力：

- 已新增统一 Agent 协作入口：
  - `POST /api/v1/agents/assist`
  - `GET /api/v1/agents/runs/{id}`
- 已形成 4 个前台可讲述的业务 Agent：
  - `KnowledgeRetrieverAgent`
  - `WorkOrderPlannerAgent`
  - `RiskControlAgent`
  - `CaseCuratorAgent`
- `/agents` 页面已从展示面板升级为正式业务协作页，支持：
  - 工单编号、设备编号、报修来源、优先级、检修模式录入
  - 场景预设
  - 图片上传
  - 知识依据锁定
  - 步骤预案展示
  - 风险提醒
  - 相似案例推荐
  - Run 回放

业务作用：

- 让多智能体能力从“技术演示”变成“业务页面”。
- 让评委能直观看到“不是一个模型直接生成答案”，而是多个角色围绕检修场景协同工作。

### 2.6 正式前端与系统工程化

已实现能力：

- 已启动 `Next.js + React + TypeScript` 正式前端工程。
- 已具备工作台、知识中心、任务中心、案例中心、历史记录、Agent 协作 6 类页面入口。
- 后端已完成一轮中等架构重组，形成：
  - `bootstrap`
  - `shared`
  - `modules`
  - `integrations`
  - `persistence`
- 已接入 Alembic、Dockerfile、GitHub Actions、部署文档与演示文档。

业务作用：

- 提升作品成品感、可维护性和交接性。
- 为正式前端继续产品化提供稳定接口和清晰结构。

### 2.7 测试与效果证据

已实现能力：

- 已有完整 `pytest` 回归测试。
- 已有软件杯评测脚本、固定评测案例和测试报告。
- 当前最新回归结果为：

`64 passed`

业务作用：

- 能说明项目不仅“能演示”，也具备一定的可验证性和稳定性。
- 方便后续答辩和版本迭代时做回归确认。

## 3. 这个项目目前的实际作用

从业务视角看，这个项目已经可以承担以下作用：

1. 帮助检修人员快速找到相关维修知识，而不是完全依赖人工翻资料。
2. 将知识检索结果转为步骤化检修预案，降低现场执行的不确定性。
3. 对关键风险进行提醒，减少直接照搬知识或跳步操作带来的问题。
4. 将一次检修过程沉淀为案例，并在审核后回流到知识库，实现经验复用。
5. 通过多智能体协作页面，把知识、流程、风险、案例四类能力组织成一个完整主线。

从比赛/答辩视角看，这个项目已经能证明：

1. 具备多模态知识检索能力。
2. 具备标准化作业辅助能力。
3. 具备案例沉淀和知识回流能力。
4. 具备多智能体协同亮点。
5. 具备正式前端、测试报告和演示主链路基础。

## 4. 当前还欠缺什么

虽然主链路已经成型，但距离“更完整的产品化”和“更强的比赛冲奖版本”仍有明显缺口。

### 4.1 知识中心仍欠缺的点

- 扫描件 PDF OCR 还没有做成专项稳定方案。
- 导入前人工校对还没做。
- 批量图片导入还没做。
- 当前导入流程还是同步处理，没有异步后台 worker。
- 来源回溯还偏“文档详情 + 前几段预览”，命中解释还不够细。

### 4.2 任务中心仍欠缺的点

- 正式前端中的任务页还比较轻，离完整业务页还有距离。
- 工单字段虽然已进入 Agent 页，但还没有完整下沉到任务主流程和任务实体中。
- 结构化检修步骤还不够完整，缺少更明确的工具、材料、参数等结构化输出。
- “检修步骤输出层”还没有完全产品化。

### 4.3 Agent 主线仍欠缺的点

- 混合检索还没做实，目前仍以当前全文检索与规则优化为主。
- rerank 还没接入。
- 相似案例推荐刚完成第一版，仍属于轻量规则召回。
- 安全提醒还偏规则化，尚未升级为更强的独立能力。
- 检修流程推荐刚有雏形，离更成熟的流程决策还差一步。

### 4.4 案例与审核仍欠缺的点

- 附件当前仍偏占位，没有完整文件上传与存储方案。
- 角色化审核界面仍未完全成型。
- 正式前端的案例页还需要进一步产品化，而不只是列表入口。

### 4.5 正式前端仍欠缺的点

- 知识页产品化较完整，但任务页、案例页、历史页仍偏轻。
- 页面之间虽已打通主叙事，但还没有完全做到“一个页面接一个页面”的强业务闭环体验。
- 角色权限、统计看板、正式导出等加分项尚未落地。

### 4.6 比赛提交物仍欠缺的点

- 虽然文档骨架已建立，但真实 PPT、视频、安装包、最终提交包还未完全收口。
- 材料与最终系统版本还需要最终一致性校对。

## 5. 当前最值得优先补的内容

如果接下来继续推进，建议按下面顺序优先处理：

1. 把任务中心和案例中心继续做成真正业务页面，不要只停在列表级入口。
2. 把工单/设备信息正式下沉到任务主流程与任务实体。
3. 完成扫描件 PDF OCR、导入前人工校对和更稳定的导入链路。
4. 为知识检索补混合检索与 rerank，继续提升命中质量。
5. 把相似案例推荐、安全提醒、流程推荐做成更强的答辩亮点。
6. 收口 PPT、视频和正式提交材料，确保比赛版本一致。

## 6. 当前结论

当前项目已经不是一个单纯的工业故障诊断原型，而是已经完成了向“设备检修知识与作业助手”的主体转型。

它已经具备：

- 知识检索
- 多模态输入
- 标准化作业辅助
- 案例沉淀与回流
- 多智能体协作展示
- 正式前端骨架
- 测试与评测基础

但它还没有完全达到“成熟产品”或“软件杯最终冲奖版”的状态。

最准确的判断是：

`主链路已经打通，成品感正在形成，核心缺口已从“有没有功能”转向“功能是否足够产品化、足够稳定、足够适合答辩和提交”。`
