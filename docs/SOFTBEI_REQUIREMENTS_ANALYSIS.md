# 软件杯需求分析文档

## 1. 项目概述

### 1.1 项目名称

`设备检修知识与作业助手`

### 1.2 赛题对应

- 赛题名称：`A基于多模态大模型技术的设备检修知识检索与作业系统`
- 赛题来源：第十五届中国软件杯
- 当前聚焦对象：`摩托车发动机检修`

### 1.3 项目定位

本项目不是通用问答系统，也不是单一故障诊断 demo，而是一套面向设备检修场景的知识检索、标准化作业指引与案例沉淀系统。现有智能分析链路作为辅助模块保留，用于增强检索后的判断与解释。

### 1.4 比赛运行环境约束

- 目标运行环境需兼容 `LoongArch` 架构 CPU 与 `银河麒麟高级服务器操作系统 V11 / V10`。
- 目标测试硬件资源仅按 `4 核 CPU + 8GB 内存 + 256GB 硬盘` 估算，不默认存在高性能 GPU。
- 因此系统必须优先保证：
  - 后端框架、数据库、Python 运行环境和依赖在国产化环境下具备可安装性或可替代方案。
  - 大模型能力以 `云端 API 稳定调用 + 流式通信` 为主，本地模型仅作为加分项或兜底能力。
  - 知识检索采用轻量架构，避免把重型向量库、图数据库作为主链路前提。

## 2. 需求背景

### 2.1 行业痛点

- 检修知识分散在手册、经验记录和口头传授中，检索成本高。
- 一线检修人员遇到故障时，往往只能凭经验判断，标准化不足。
- 大模型直接回答虽快，但缺少知识出处和作业闭环，难以直接用于实际检修。
- 一次检修的经验难以结构化沉淀，导致相似问题反复排查。

### 2.2 建设目标

- 让检修人员能通过文本、设备型号、故障图片快速检索到相关知识。
- 将知识检索结果转化为标准化作业步骤，而不仅是文本建议。
- 将每次检修案例沉淀为可审核、可复用的新知识。
- 形成“检索 -> 作业 -> 沉淀 -> 回流”的闭环。
- 在受限国产化环境下保持系统可部署、可演示、可复现，不依赖高性能 GPU 或重型中间件。
- 建立统一模型接入层，保证云端模型可平滑切换，并为轻量本地模型保留最小扩展位。

## 3. 用户与场景

### 3.1 用户角色

- 检修技师：发起知识检索、执行检修任务、提交案例。
- 审核人员：复核案例内容、修正总结、审核入库。
- 演示评委：快速理解系统价值，关注功能完整性、稳定性和实用性。

### 3.2 核心场景

- 场景 A：设备出现启动困难、异响、渗漏等故障时，检修技师快速检索知识。
- 场景 B：根据检索结果自动生成标准化作业指引并执行。
- 场景 C：检修完成后录入实际处理过程和结果，进入审核与回流。

## 4. 功能需求

### 4.1 知识检索

- 支持文本、设备型号、单张故障图片作为输入。
- 检索结果需返回命中知识条目、来源文件、章节或页码。
- 支持通用手册与具体型号知识共同参与检索。

### 4.2 标准化作业指引

- 根据检索结果生成检修任务。
- 任务包含步骤编号、操作说明、风险提示、注意事项和完成确认。
- 每一步都能回溯到知识来源。

### 4.3 案例沉淀与审核

- 支持录入故障描述、处理步骤、处理结果、附件信息。
- 支持人工修正检索结果和总结。
- 审核通过后自动回流入知识库。

### 4.4 智能分析辅助

- 在知识检索和作业流程之外，提供大模型辅助分析。
- 该模块不是主产品入口，只作为辅助说明与排查建议来源。

### 4.5 模型接入与硬状态机控制

- 模型接入需统一封装，支持不同云端模型或多模态模型平滑切换。
- 检修流程中的关键步骤需支持硬性状态机控制，而不是完全依赖自由生成。
- 对高风险步骤，系统必须能够强制要求提供“证据”后再进入下一步，例如：
  - 上传检修确认照片
  - 满足特定知识依据锁定条件
  - 满足遥测或人工确认条件
  - 完成人工授权
- 系统需明确给出“可继续 / 待确认 / 被拦截”的状态，而不是仅返回文字提示。

## 5. 非功能需求

### 5.1 国产化兼容性

- 核心后端、数据库、部署脚本和运行时依赖应能在 `LoongArch + 银河麒麟` 环境下执行或给出替代方案。
- 不应将只能在高性能 x86 + GPU 环境运行的方案作为主交付路径。

### 5.2 资源约束与性能

- 系统主链路需能在 `4 核 CPU + 8GB 内存 + 256GB 硬盘` 的受限环境下稳定运行。
- 检修主流程不得依赖本地部署 7B/8B 级大模型作为前提。
- 大模型能力优先采用云端调用，本地仅保留轻量补充方案。
- 数据层优先采用 `PostgreSQL + 全文检索 + 轻量 rerank`，不将重型向量库或图数据库作为主要求。

### 5.3 用户体验与界面

- 软件界面需合理、美观大方。
- 人机交互需便捷，适合评委和检修人员快速理解与操作。
- 评委应在 3 到 5 分钟内看懂主链路，并能区分“知识检索、作业执行、案例回流、智能辅助”四类功能。

### 5.4 可追溯性与可观测性

- RAG 结果必须具备知识锚点，能够溯源到原始手册或知识文档的具体位置。
- 作业步骤、案例回流和审核结果都要保留来源依据。
- 后台需具备统一错误码、Request ID 和基础指标日志，便于排障和部署验收。

### 5.5 演示与部署稳定性

- 固定案例和固定检索词下需稳定命中。
- 需支持 PostgreSQL + FastAPI 的独立部署。
- 需具备明确的服务器部署 runbook、迁移流程和健康检查方式。

## 6. 边界与排除项

- 当前版本不追求完整工业协议接入。
- 当前版本不做复杂角色权限体系。
- 当前版本不以“本地独立运行 7B/8B 大模型”作为主交付要求。
- 当前版本不引入 `Milvus / Neo4j / GraphRAG / RLHF / DPO` 等高复杂度平台作为主链路前提。
- 当前版本不做视频、语音等更重的多模态扩展。

## 9. 当前阶段结论

当前项目已完成软件杯核心主链路，但正式提交前必须继续围绕三件事收口：

- 在国产化受限环境下保证主链路可运行、可部署、可解释
- 在需求层明确“轻量技术选型、云端模型优先、硬状态机证据门禁”的约束
- 将文档、PPT、视频、部署包和测试报告一起视为正式交付物

该文档作为需求层源文件，用于后续功能设计、产品说明书、PPT、视频和部署说明保持一致。
