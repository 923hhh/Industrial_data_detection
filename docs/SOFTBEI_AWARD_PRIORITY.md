# 软件杯冲奖优先级清单

## Summary

当前项目已经具备“可演示、可部署、可测试”的比赛原型基础，下一阶段不应再平均发力，而要围绕评委最直接感知到的得分项集中投入。当前固定优先顺序为：

`材料完整度 > 演示稳定性 > 知识库质量与检索效果 > 前端成品感 > 额外工程化加分`

## 必须做

- 完成 `TODO-SB-8` 全部提交材料：需求分析、功能设计、产品说明书、测试报告、安装部署文档、PPT、演示视频、源码包必须一套齐全且互相一致。
- 固定一条 5 分钟演示主链路：`知识检索 -> 作业指引 -> 步骤执行 -> 案例沉淀 -> 审核回流`，所有页面、话术和截图都围绕这条主链。
- 把服务器上的代码、依赖、数据库配置和本地最新版完全同步，避免“本地能跑、服务器还是旧逻辑”。
- 把知识库内容做实：继续导入真实维修手册、标准步骤和典型案例，至少保证评委演示时使用的检索词稳定命中。
- 优化知识检索命中率：重点解决“长中文故障描述”“通用手册 vs 具体型号过滤”“同义词不一致”这三类问题。
- 准备固定演示检索词清单、固定任务清单、固定案例清单，避免现场临时输入导致空结果。
- 对正式工作台做最后一轮收口：保证首页、检索页、任务页、案例页视觉一致，主按钮、状态提示、空结果提示统一。
- 完成最终回归验收：接口、前端主链路、服务器部署链路、知识检索主案例都要重新跑一遍，并留截图或录屏证据。

## 应该做

- 继续打磨已落地的“通用手册可命中具体型号”检索策略，避免不同知识源的型号过滤再次产生空结果。
- 继续扩展已落地的检修术语同义词、query rewrite 和关键词扩展规则，例如“动力下降/功率下降”“点火异常/火花塞问题”等。
- 把 PDF 导入流程产品化到页面或管理接口，至少做到“导入后可预览、可确认、可重复导入不脏数据”。
- 增加 OCR 能力，让扫描件或图片型 PDF 也能进入知识导入链路。
- 把图片上传从联调能力升级为正式知识中心能力，分别支撑故障图检索和文档导入。
- 增加多模态问答能力，让知识中心可以围绕“文本 + 型号 + 图片”返回带引用的回答，而不只是命中列表。
- 在任务链路中补齐工单/设备信息字段，使任务、案例和历史记录更接近真实业务对象。
- 补“检修步骤输出”层，让知识检索结果能稳定转成结构化步骤、工具/材料清单和确认动作。
- 把案例审核页做得更像正式评审界面，而不是仅能联调的静态页。
- 补图片检索的自动化验证，目前评测仍以“文本 + 型号”为主，和当前已具备的图片 fallback / 多模态入口之间仍有落差。
- 把测试报告里的指标解释写得更像答辩材料，而不是只保留工程日志口径。
- 补一套答辩问答：为什么要用多模态、为什么要做知识回流、为什么你的方案优于直接问大模型。
- 整理一页系统架构图、一页业务流程图、一页数据库关系图，直接服务 PPT 和答辩。

## 加分项

- 给知识检索增加 rerank 或更强的 query rewrite，让复杂故障叙述更稳定命中。
- 增加 OCR 或扫描件导入能力，让知识导入不只适用于可提取文本 PDF。
- 增加上传文件存储、附件查看和案例附件回溯能力。
- 增加混合检索，把全文检索、规则召回和案例召回组合起来，提高复杂工况下的稳定命中率。
- 增加相似案例推荐，让知识命中之外还能展示历史处理经验和结果对照。
- 将安全提醒升级为独立动态能力，按设备、步骤和风险点自动生成。
- 增加检修流程推荐，让 Agent 不只输出单步建议，而是给出适合当前工况的整条检修路径。
- 补 Nginx、HTTPS、结构化日志、异常追踪和更完整的线上监控。
- 增加最小角色权限，例如检修人员、审核人员、管理员三个角色。
- 增加更像产品的统计看板，如知识命中次数、案例通过率、任务完成率。
- 增加导出为 PDF 或 Word 的正式作业单，而不仅是 JSON 或摘要文本。

## 固定演示资产

- 固定设备对象：`摩托车发动机 / LX200`
- 固定知识检索词：`火花塞`、`冷启动困难`、`正时链条`、`机油渗漏`、`温度偏高`
- 固定完整任务链路：
  - `SB-01 启动困难积碳复盘案例`
  - `SB-04 正时链条异响复盘案例`
  - `SB-06 缸盖渗漏修复复盘案例`

## 验收标准

- 评委在 3 到 5 分钟内能完整看懂产品价值，不需要长时间解释系统结构。
- 至少 5 组固定检索词和 3 条完整任务链路能稳定跑通，不出现空结果或随机失败。
- 服务器演示环境与本地代码版本一致，且重启后服务可自动恢复。
- 所有正式提交材料与最终系统版本一致，不出现“文档写一套、系统演示另一套”。
- PPT、视频、工作台页面和测试报告都围绕同一条主叙事，不再出现“工业故障诊断原型”和“软件杯检修系统”两套口径混用。

## Assumptions

- 目标是软件杯冲高奖项，而不是继续扩展通用工程功能。
- `LoongArch + 银河麒麟` 暂不纳入当前主线，但正式提交前若赛题硬性要求仍需补回。
- 现阶段最重要的增量不是新后端模块，而是“知识质量、演示稳定性、材料完整度”。
- 当前诊断模块继续保留为辅助分析能力，不再回到主产品叙事中心。
