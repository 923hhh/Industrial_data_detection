# 软件杯功能测试报告（TODO-SB-7）

## 1. 报告目的

本报告用于为软件杯赛题《基于多模态大模型技术的设备检修知识检索与作业系统》提供可复核的阶段性证据。当前目标不是学术算法论文式评测，而是证明现有系统已经具备：

- 固定知识源上的稳定知识检索能力
- 可引用出处的检修作业闭环
- 审核通过后的案例回流能力
- 对比旧版能力和弱基线时的明确优势

## 3. 评测样本

- 知识文档：6 份
- 标准案例：12 个
- 场景覆盖：
  - 成功场景：5 个
  - 模糊场景：3 个
  - 失败场景：4 个
- 回流案例：4 个

## 4. 指标定义

- 检索命中率：案例是否命中预期知识条目或预期检修术语
- 引用覆盖率：检索命中的结果是否带有来源文件和章节/页码
- 作业闭环完成率：命中案例是否能完成“建任务 -> 全步骤完成 -> 导出摘要”
- 案例回流召回率：审核通过案例是否能在后续检索中重新命中

## 5. 基线定义

### 当前系统

- `POST /api/v1/knowledge/search`
- `POST /api/v1/tasks`
- `PATCH /api/v1/tasks/{id}/steps/{step_id}`
- `POST /api/v1/cases`
- `POST /api/v1/cases/{id}/review`

### 关键词检索基线

- 仅对种子文档做简单关键词匹配
- 不支持出处引用结构化返回
- 不支持作业闭环
- 不支持案例审核回流

### 无知识引用的大模型直接回答

- 不做知识检索
- 不做结构化引用
- 不做作业步骤和案例回流

### 旧版诊断控制台

- 仅适合时间窗诊断子模块
- 不覆盖赛题要求的知识检索、作业闭环和知识沉淀

## 7. 本轮评测发现与已修复问题

本轮自动评测直接暴露了两个真实缺陷，并已在代码中修复：

1. `SQLite fallback 检索过于依赖整句匹配`
   - 现象：自然语言查询在本地/测试环境下容易直接返回空结果
   - 修复：在 [knowledge_service.py](/e:/南京航空航天大学/aaa大创/智能体案例/dachuang_project/app/services/knowledge_service.py) 中补充分词匹配与元数据兜底检索

2. `案例审核入库触发 async lazy-load`
   - 现象：审核通过时访问 `document.chunks` 触发 `MissingGreenlet`
   - 修复：在 [case_service.py](/e:/南京航空航天大学/aaa大创/智能体案例/dachuang_project/app/services/case_service.py) 中改为显式删除旧分段并 `add_all` 新分段，避免异步惰性加载

3. `长中文故障描述在正式检索环境下命中不稳定`
   - 现象：直接使用长段自然语言查询时，即使知识库中存在相关手册，也可能因中文全文检索粒度问题返回空结果
   - 修复：在 [knowledge_service.py](/e:/南京航空航天大学/aaa大创/智能体案例/dachuang_project/app/services/knowledge_service.py) 中增加 query rewrite 与 `effective_keywords`，先将故障描述收敛为稳定检修术语，再进入检索主链

4. `图片 fallback 依赖中文文件名，英文文件名场景不稳定`
   - 现象：现场上传 `spark-plug`、`timing-chain`、`oil-leak` 这类英文命名图片时，兜底检索线索不足
   - 修复：在 [image_analysis_service.py](/e:/南京航空航天大学/aaa大创/智能体案例/dachuang_project/app/services/image_analysis_service.py) 中增加英文文件名到中文检修术语的映射
