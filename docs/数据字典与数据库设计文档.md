## 1. 文档目的

本文档定义「工业多模态检修系统」首期 MVP 的**逻辑数据模型、物理表结构、字段字典、枚举与约束、索引与分库分表策略（首版无分表）**，以及与 **向量检索侧** 的元数据对应关系。实现须与《系统架构文档》V1.1、《MVP 产品需求文档》V1.2（尤其 **§2.6 拍板**）、《检修业务流程与 SOP 文档》V2.1（工单状态 **S1～S12、SX**）保持一致。

本文档为《接口文档》中请求/响应字段与《验收与测试文档》**V1.1**（`docs/验收与测试文档.md`，含 §6.1 数据库断言）中数据校验提供依据（**`docs/接口文档.md` V1.1** 已与本字典对齐）。

## 3. 技术约定

| 项 | 约定 |
| -- | ---- |
| 命名 | 关系库对象统一 **snake_case**；表名复数或业务名单词本文已统一如下 |
| 主键 | 首版推荐 **`BIGINT` 自增**（SQLite `INTEGER`）；若使用 UUID，须在《接口文档》全链路统一 |
| 时间 | **`created_at` / `updated_at`**：`TIMESTAMP WITH TIME ZONE`（PostgreSQL）或 SQLite 存 UTC ISO 文本 + 应用层解析 |
| 删除 | 首版 **物理删除** 慎用；配置类与知识类建议 **`status` + 下架时间** 软删；审计表只增不改 |
| 字符集 | UTF-8 |
| JSON | PostgreSQL `JSONB` / SQLite `TEXT` 存 JSON 字符串；应用层 Pydantic 校验 |
| 迁移 | **Alembic** 版本化；开发与演示 **SQLite**，目标验收 **PostgreSQL** 须通过同一套迁移脚本验证 |

## 5.1 约束落地策略：数据库强约束 vs 应用层保证

| 规则 | 数据库层（推荐迁移中显式声明） | 应用层（仍须校验，防绕过） |
| ---- | ------------------------------ | ---------------------------- |
| 同工单仅一条「进行中」升级（`open`/`in_progress`） | **部分唯一索引** `UNIQUE(work_order_id) WHERE status IN ('open','in_progress')`（PostgreSQL；SQLite 3.31+ 支持，见 §5.2） | 创建前 SELECT 校验；返回 **409** `ESCALATION_IN_PROGRESS`（MVP FR-ESC-04） |
| 同工单仅一条 `is_latest=true` 的回填 | **部分唯一索引** `UNIQUE(work_order_id) WHERE is_latest = true` | 事务内先将旧行 `is_latest=false` 再插入新行；禁止并发双 true |
| `resolution_status=unresolved` 时 `post_unresolved_action`、`unresolved_reason_code` 非空 | **`CHECK` 约束**（见 §6.12 后附 SQL 语义） | 请求体验证与 **400** 字段错误（MVP §2.6） |
| `resolved` 且 `closure_code=OTHER` 时 `detail_notes` 非空 | **CHECK** 或仅应用层 | 首版可仅用应用层 **400**；若上 CHECK 须与接口完全一致 |
| 同工单 + 同工步仅一条 `pending` 审批 | **部分唯一索引** `UNIQUE(work_order_id, step_no) WHERE status = 'pending'` | 创建审批任务前查重 |
| 幂等审批（同结论重复提交） | **不新增第二行终态**；依赖 `UNIQUE` 无帮助，靠 **应用层事务 + 行锁** 与 MVP §2.6.3 | 返回 **200 + ALREADY_PROCESSED** |
| 枚举合法值 | **CHECK** 或仅应用层 | 首版最小可用可只做应用层；**推荐 PostgreSQL 目标环境上 CHECK**，SQLite 演示环境可省略以简化 Alembic 分支 |

**原则**：能在 **PostgreSQL 验收环境** 落地的脏数据防护优先写进迁移；SQLite 开发环境若语法差异大，须在 `alembic` 或 README 中说明「演示库弱约束、麒麟库强约束」，并在《验收与测试文档》中区分环境用例。

## 5.2 版本与「当前有效」唯一性（推荐索引 / 约束）

以下为避免 **多 published、多 pending 审批、双 latest 回填** 等脏数据；**具体 DDL 以 Alembic 为准**，本文给出语义。

| 对象 | 规则 | 推荐实现 |
| ---- | ---- | -------- |
| `flow_templates` | 同一 `(device_type, maintenance_level)` 下 **至多一条** `status = 'published'` | `CREATE UNIQUE INDEX ux_ft_pub ON flow_templates(device_type, maintenance_level) WHERE status = 'published'` |
| `knowledge_articles` | 同一 **知识系列** 在任意时刻 **至多一条** `status = 'published'` | 增加 `series_id BIGINT NOT NULL`（首条创建时 `series_id = id`；新版本沿用同 `series_id`）；`CREATE UNIQUE INDEX ux_kb_pub ON knowledge_articles(series_id) WHERE status = 'published'`；旧版本发布前须先置 `archived` 或 `pending_publish` 流转 |
| `approval_tasks` | 同一 `(work_order_id, step_no)` **至多一条** `status = 'pending'` | 见 §5.1 部分唯一索引 |
| `escalations` | 同一 `work_order_id` **至多一条** `status IN ('open','in_progress')` | 见 §5.1 |
| `work_order_fillings` | 同一 `work_order_id` **至多一条** `is_latest = true` | 见 §5.1 |

## 7. 向量检索侧数据约定

向量库存储 **embedding 与向量**；业务库须维护可join的元数据（以下二选一，**首版拍板：业务表 `kb_chunks` + 向量库 `chunk_id` 一致**）。

### 7.1 `kb_chunks`（推荐）

| 字段名 | 类型 | 说明 |
| ------ | ---- | ---- |
| `id` | BIGINT PK | 与向量库 document id 一致 |
| `source_type` | VARCHAR(32) | NOT NULL | **`manual`** 官方/设备手册 \| **`sop`** 结构化 SOP 导出 \| **`case`** 历史案例 \| **`kb_article`** 已发布知识条目切片 |
| `source_ref` | VARCHAR(256) | 可空 | 手册文件名、SOP 业务 id、`knowledge_articles.id` 等 |
| `source_document` | VARCHAR(512) | 可空 | 展示用文档名 |
| `position` | INT | 段序号 |
| `text` | TEXT | 全文片段 |
| `metadata_json` | JSONB | 设备类型、章节等 |
| `embedding_model` | VARCHAR(64) | 嵌入模型名 |
| `is_active` | BOOLEAN | NOT NULL，默认 true | **下架/回滚**时置 false；检索默认 `WHERE is_active`；避免全量重建向量库 |

## 8. 索引与性能（首版）

- 所有外键列建 **BTree 索引**（SQLite 亦支持）。
- **部分唯一索引**（§5.2）为防脏数据关键路径，须在 **PostgreSQL 验收环境** 的迁移中启用；SQLite 开发环境若版本不足，须在 README 标注「弱约束环境」。
- `work_order_events`、`audit_logs` 数据增长快，演示环境可定期归档；生产按 **时间分区** 为后续演进（首版不强制）。
- 大文本 `TEXT` 不参与组合索引。
