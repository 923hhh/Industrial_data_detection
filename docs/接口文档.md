## 1. 文档目的

本文档定义「工业多模态检修系统」首期 MVP 的 **HTTP API 契约**（路径、方法、鉴权、请求/响应体字段、业务错误码、幂等与状态迁移约定），供前后端联调、OpenAPI 生成与《验收与测试文档》（`docs/验收与测试文档.md`）编写用例。**当前文档版本：V1.1**。

**实现栈约定**：与《系统架构文档》一致，服务端为 **FastAPI**；检修业务 API 与既有传感器诊断 API **路径前缀隔离**。

## 3. 全局约定

### 3.1 Base URL 与版本前缀

| 项 | 值 |
| -- | -- |
| 开发默认 | `http://localhost:8000` |
| 检修业务 API 前缀 | **`/api/v1/maintenance`** |
| 既有诊断示例（非本域） | `/api/v1/diagnose`、`/api/v1/diagnose/stream` 等，**不得与本文路径混用同一业务语义** |

全文路径写作：`{BASE}{PREFIX}/...`，其中 `{PREFIX}=/api/v1/maintenance`。

### 3.2 传输与格式

- **HTTPS**（生产）；本地可 HTTP。
- **Content-Type**：`application/json`，除非上传接口显式为 `multipart/form-data`。
- **字符编码**：UTF-8。
- **时间**：ISO 8601，带时区，如 `2026-04-11T12:00:00+08:00`。

### 3.3 认证

| 项 | 约定 |
| -- | -- |
| 机制 | **Bearer Token**（JWT Access Token），请求头：`Authorization: Bearer <access_token>` |
| 获取令牌 | `POST {PREFIX}/auth/login`（见 §4） |
| 未带令牌或无效 | **401** `UNAUTHORIZED` |
| 令牌有效但无接口权限 | **403** `FORBIDDEN` |

### 3.4 统一响应包（业务体）

成功时 HTTP **2xx**，响应 JSON 推荐结构：

```json
{
  "success": true,
  "data": {},
  "business_code": null,
  "message": null
}
```

失败时 HTTP **4xx/5xx**；**业务可预期降级**（空命中、低置信、模型不可用）**统一**为 HTTP **200** + `success:false` + `business_code`（见 **§3.4.1**）。**真异常**（未捕获代码错误）用 **5xx**。推荐结构：

```json
{
  "success": false,
  "data": null,
  "business_code": "VALIDATION_ERROR",
  "message": "人类可读摘要",
  "errors": [
    { "field": "closure_code", "code": "INVALID_ENUM", "message": "可选字段级说明" }
  ]
}
```

- **`business_code`**：大写下划线枚举字符串，与 HTTP 状态组合可唯一定位问题。
- **`errors`**：可选；校验类 **400** 建议使用。

### 3.4.1 HTTP 状态与 `business_code` 边界（首版拍板）

| 情形 | HTTP | `success` | `business_code` |
| ---- | ---- | ----------- | --------------- |
| 请求合法且业务成功 | 2xx | true | null 或省略 |
| 参数非法、鉴权失败、状态冲突、资源不存在 | **4xx** | false | 与 §18 一致，如 `VALIDATION_ERROR`、`INVALID_STATE_TRANSITION` |
| 检索无命中、低置信、模型降级等 **已处理** 的「软失败」 | **200** | false | `EMPTY_HIT`、`LOW_CONFIDENCE`、`MODEL_UNAVAILABLE` 等 |
| 未捕获服务端异常 | **5xx** | false | `INTERNAL_ERROR`（实现定义） |

**禁止**：同一类软失败混用 200 与 4xx，前端只依赖 `HTTP + success + business_code` 三分支即可。

### 3.5 分页（列表类）

查询参数统一支持（未列出的列表接口默认也适用）：

| 参数 | 类型 | 默认 | 说明 |
| ---- | ---- | ---- | ---- |
| `page` | int | 1 | ≥1 |
| `page_size` | int | 20 | 1～100，超出 **400** |

**硬约束**：凡分页列表接口，成功时 `data` **必须且只能**为以下结构（**不得**使用 `records`、`list`、`pagination` 等变体）：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

### 3.6 幂等与重复提交

- **审批提交**（§10）：遵守《MVP 产品需求文档》**§2.6.3**（`ALREADY_PROCESSED` / `CONFLICT`）。
- **升级创建**：依赖数据库唯一约束，重复活跃升级 → **409** `ESCALATION_IN_PROGRESS`。
- 可选请求头 **`Idempotency-Key`**：`POST` 创建类接口若实现，须与《数据字典》审计对齐（首版可不实现，实现则文档化 TTL）。

### 3.7 首版实现拍板（消除「择一」）

| 项 | 拍板结论 |
| -- | -------- |
| `POST .../auth/logout` | **仅客户端丢弃 token**；服务端不维护黑名单（首版简化）。 |
| 附件下载 | **`GET .../attachments/{id}/content` 返回 302**，`Location` 为 **短时签名 URL**（如 300s），对象存储或应用内签名生成器实现；**不采用**直连 200 流（便于权限收口）。 |
| 单文件超限 | 统一 **HTTP 413**，`business_code`：`PAYLOAD_TOO_LARGE`（不再使用 400 变体）。 |
| 工步确认时高危未批 / 状态不允许 | 统一 **HTTP 409**，`business_code`：`STEP_NOT_ALLOWED`（不使用 403）。 |
| 审批 `resolve` 请求体终态字段名 | 与表 `approval_tasks.status` 一致，字段名统一为 **`status`**，取值：`approved` \| `rejected` \| `need_more_info`（与数据字典 §4.5 一致）。 |

## 8. 工单与检索（核心链）

### 8.0 检索、快照与消息的关系（首版拍板）

| 规则 | 说明 |
| ---- | ---- |
| 一次 `POST .../retrieval` | **必须** 产生 **恰好一条** `assistant` 角色的 `work_order_messages` 记录，且 **`retrieval_snapshot_id` 与该 message 1:1 绑定**（同一次调用写入同一事务）。 |
| `retrieval_snapshots` 与 message | 快照表 **`work_order_id`** 冗余合法；**对外展示以 `message_id` 为主键**做会话与标注。 |
| 用户 `POST .../messages` | 在**同一工单上下文**追加 `user` 消息；下一次 `retrieval` **携带**最新用户问题 + 历史摘要策略由实现定，**须在 OpenAPI 描述写清默认取最近 N 条 user 消息**。 |
| `POST .../escalations` 的 `related_message_id` | **可选**；若传则 **必须为**本工单下 **`role=assistant`** 的 `message_id`（用于指向引发升级的模型回答），否则 **400** `INVALID_MESSAGE_REF`。 |

### 8.4 工单状态迁移（服务端强校验）

**原则**：客户端 **不直接写 `status` 字符串**；**首版拍板：语义化动作接口**（更安全、易测）。

**升级会诊创建**：优先使用 **§11 `POST .../work-orders/{id}/escalations`**（创建升级记录并迁移工单状态）。**`POST .../actions/request-escalation`** 为 **等价别名**：请求体、校验、幂等与 §11 创建接口 **完全相同**，便于前端按「动作」语义命名；实现可路由到同一服务。

| 方法 | 路径 | 说明 |
| ---- | ---- | ---- |
| POST | `{PREFIX}/work-orders/{work_order_id}/actions/request-escalation` | 与 **§11 创建升级** 等价（见上） |
| POST | `{PREFIX}/work-orders/{work_order_id}/actions/enter-maintenance` | 进入检修 **→S7**（见下表） |
| POST | `{PREFIX}/work-orders/{work_order_id}/actions/complete-maintenance` | **S7→S8** |

非法迁移统一 **409**，`business_code`：**`INVALID_STATE_TRANSITION`**。

#### 8.4.1 状态前置条件表（工单 `status`，与《检修业务流程与 SOP 文档》§4.1 对齐）

| 接口 / 动作 | 允许当前工单 `status` | 成功后工单 `status`（典型） | 备注 |
| ----------- | --------------------- | --------------------------- | ---- |
| `POST .../escalations` 或 `actions/request-escalation` | **S2、S3** | **S4** | 与「未命中/主动升级」一致；**S2** 多为检索软失败后一线确认升级 |
| `POST .../escalations/{id}/resolve`（专家结论） | **S4、S5**（且升级单状态允许） | **S5、S6、S7** 或退回 **S3** 等 | 精确分支与 SOP 高危/非高危一致，见 **§11.1** |
| `POST .../actions/enter-maintenance` | **S3、S5** | **S7** | **不含** `S6→S7`（须走 **§10 审批通过**） |
| `POST .../approval-tasks/{id}/resolve`（`status=approved`） | 工单 **S6** 且任务绑定匹配 | **S7** | 审批域接口，见 §10 |
| `POST .../actions/complete-maintenance` | **S7** | **S8** | — |
| `POST .../fillings` | **S8** | **S9** | 见 §9 |
| `POST .../steps/confirm` | 见 §12 | **不变**（推进工步，不单独改工单状态） | 工步与状态并行，见 §12 |

OpenAPI 与各接口 `description` 须引用本表；《验收与测试文档》可直接抄用。

`status` 取值：`approved` \| `rejected` \| `need_more_info`；与表 **`approval_tasks.status`** 及数据字典枚举 **一致**（§3.7）。

**成功 `data` 示例**

```json
{
  "id": 88,
  "work_order_id": 1001,
  "status": "approved",
  "resolved_at": "2026-04-11T14:00:00+08:00",
  "work_order": { "id": 1001, "status": "S7" }
}
```

| HTTP | business_code | 说明 |
| ---- | ------------- | ---- |
| 200 | — | 首次处理成功 |
| 200 | `ALREADY_PROCESSED` | 幂等重复提交（与 §2.6.3 一致） |
| 409 | `CONFLICT` | 终态与已存记录冲突 |
| 409 | `INVALID_STATE_TRANSITION` | 工单非 S6 或任务不匹配 |

**幂等（§2.6.3）**：重复相同结论 → **200** + `ALREADY_PROCESSED`。

## 15. 审计与系统配置

| 方法 | 路径 | 说明 | 角色 |

### 15.1 `GET .../admin/audit-logs`

**查询参数（首版最小）**

| 参数 | 说明 |
| ---- | ---- |

| `resource_type` | 如 `work_order`、`escalation`、`knowledge_article` |

```json
{
  "id": 9000,
```

### 15.2 `GET .../admin/system-configs`

- `is_sensitive=true` 的配置项：返回 **`value_masked`**（如 `****last4`）或 **省略 `value`**，**不得**返回明文密钥。
- 响应可为 **不分页列表** 或 **`{ items, total, page, page_size }`**；若分页则须符合 §3.5。

### 15.3 `PATCH .../admin/system-configs/{key}`

- **成功**：**HTTP 200**，`success:true`，`data` 为 **更新后的完整配置对象**（含 `key`、`value` 或脱敏字段、版本号），**不使用 204**（便于前端刷新）。
- 校验失败 **400**；键不存在 **404**。

## 18. `business_code` 清单（首版最小集）

| business_code | 典型 HTTP | 含义 |
| --------------- | --------- | ---- |
| `UNAUTHORIZED` | 401 | 未登录或令牌无效 |
| `FORBIDDEN` | 403 | 无角色权限 |
| `NOT_FOUND` | 404 | 资源不存在 |
| `DEVICE_NOT_FOUND` | 404 | 创单时设备不存在 |
| `VALIDATION_ERROR` | 400 | 字段校验失败 |
| `INVALID_STATE_TRANSITION` | 409 | 工单 / 工步 / 知识条目等状态不允许 |
| `ESCALATION_IN_PROGRESS` | 409 | 重复活跃升级 |
| `EXPERT_NOT_CONFIGURED` | 400 | 设备无责任专家 |
| `INVALID_MESSAGE_REF` | 400 | `related_message_id` 非法 |
| `ALREADY_PROCESSED` | 200 | 审批、工步重复确认等幂等命中 |
| `CONFLICT` | 409 | 审批结论冲突 |
| `STEP_NOT_ALLOWED` | 409 | 工步确认不满足前置 |
| `SERIES_PUBLISHED_CONFLICT` | 409 | 同 `series_id` 已存在已发布条目 |
| `PAYLOAD_TOO_LARGE` | 413 | 上传超限 |
| `EMPTY_HIT` / `MODEL_UNAVAILABLE` / `LOW_CONFIDENCE` | **200** | 检索软失败，与 **§3.4.1** 一致 |
