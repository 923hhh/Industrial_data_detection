# 软件杯后续技术改进优先级

## 1. 文档目的

本文用于整理基于外部技术分析文档与当前仓库实现现状得到的后续改进建议，重点回答两件事：

- 接下来什么值得做
- 现阶段什么不值得优先做

本文默认目标不是把项目做成商用工业平台，而是在软件杯场景下，继续提升系统的技术说服力、演示完整度和工程合理性。

## 2. 判断依据

- 外部参考文档强调的方向：
  - 从 `RAG 检索` 升级到 `可执行 Agent`
  - 从 `文本回答` 升级到 `工具调用 + 合规校验 + 人类在环`
  - 从 `一次性生成` 升级到 `知识补丁化更新 + 版本化演进`
  - 从 `阻塞式交互` 升级到 `流式事件 + 渐进式反馈`
  - 从 `单一模型调用` 升级到 `本地 / 云端统一模型网关`
- 当前仓库已具备的基础：
  - 已有知识导入、OCR、知识检索、任务、案例、审核回流、正式前端链路
  - 已有 Agent run 持久化、知识导入异步任务化、请求 ID、统一错误码、指标日志
  - 已有 PostgreSQL 检索优化、rerank、主链路集成测试
  - 已有诊断 SSE 流式接口，但正式 `/agents` 主链路还不是完整事件流架构

### 2.1 软件杯比赛约束补充

- 目标运行环境需考虑 `LoongArch + 银河麒麟 V11 / V10`
- 目标测试硬件按 `4 核 CPU + 8GB 内存 + 256GB 硬盘` 估算
- 不默认存在高性能 GPU，因此不应将本地 7B/8B 模型部署视为主链路前提
- 轻量化技术选型优先级高于“堆叠更多平台名词”
- 比赛最终交付标准不仅包含代码，还包含需求、设计、测试、部署、PPT 和视频材料

## 3. 总体结论

当前最值得投入的方向，不是继续平铺更多页面或堆叠更多模型名词，而是把系统继续从“会给建议”推进到“会按规则调用工具、会阻断高风险动作、会留下结构化证据链、会在受限国产化环境下稳定运行”。

在当前比赛约束下，固定优先顺序建议修正为：

`P0 轻量化可运行闭环 > P1 统一模型网关与硬状态机 > P2 知识演化与多模态增强`

## 4. 建议优先做

### P0：必须优先做

#### 4.1 Agent 工具注册表与执行审计

- 目标：把 Agent 从“生成检修建议”升级成“可调用正式工具的作业引擎”。
- 建议内容：
  - 建立统一工具注册表，所有工具定义固定 `name / purpose / input schema / output schema / risk level`
  - 第一批优先落地：
    - `query_device_telemetry`
    - `fetch_historical_repairs`
    - `require_human_authorization`
    - `validate_safety_preconditions`
  - 每次工具调用都记录到 Agent run 中，形成可回放的执行链
  - 工具失败、超时、拒绝执行都要有结构化状态
- 业务价值：
  - 让系统从“回答型助手”变成“可执行作业系统”
  - 方便答辩时展示 Agent 不是空口生成，而是在调用真实业务能力

#### 4.2 合规校验与安全拦截规则层

- 目标：让关键检修步骤在推进前满足明确的机器可校验条件。
- 建议内容：
  - 为任务步骤补充前置条件字段，例如：
    - `temperature_below`
    - `power_isolated`
    - `ppe_verified`
    - `requires_manual_approval`
  - 增加规则校验服务，在步骤执行前做自动拦截或提醒
  - 对高风险步骤统一要求人工确认并留痕
- 业务价值：
  - 明显提升“工业场景可信度”
  - 比继续堆生成式能力更能体现系统不是聊天机器人

#### 4.3 层级化知识锚点与可定位检索

- 目标：让检索结果不只返回文档片段，还能定位到“哪一章、哪一步、哪张图”。
- 建议内容：
  - 在知识导入时保留章节层级、步骤编号、图片引用、页码锚点
  - 检索结果增加：
    - `section_path`
    - `step_anchor`
    - `image_anchor`
    - `document_page`
  - 前端支持从命中结果直接跳到对应章节与分段
- 业务价值：
  - 提高“精准语义检索与跨模态匹配”的可解释性
  - 让评委能看见系统不是只命中一段文本，而是真正定位知识源

#### 4.4 正式主链路事件流化

- 目标：把流式交互从诊断子模块扩展到正式业务页面。
- 建议内容：
  - `/agents` 改造成事件流驱动页面，展示：
    - 检索中
    - 工具调用中
    - 风险校验中
    - 等待人工授权
    - 任务预案生成完成
  - 知识导入任务页也展示异步处理阶段与失败原因
  - 前端使用统一事件状态组件，而不是每个页面各自维护 loading 文案
- 业务价值：
  - 大幅提升演示效果和“智能体正在工作”的真实感
  - 降低等待黑盒感

#### 4.5 评测集与业务指标固化

- 目标：让检索和 Agent 能力不只靠主观描述，而能量化比较。
- 建议内容：
  - 建立固定评测集，至少覆盖：
    - 设备型号精准命中
    - 同义故障描述命中
    - 图片型故障命中
    - 案例回流后二次命中
    - 高风险步骤拦截
  - 固定输出指标：
    - `Top1 / Top3 命中率`
    - `同型号命中率`
    - `案例回流命中率`
    - `工具调用成功率`
    - `人工授权触发率`
- 业务价值：
  - 便于答辩展示“做了什么提升”
  - 为后续 rerank、规则和工具调用优化提供基准

#### 4.6 国产化运行环境验证与轻量化选型固化

- 目标：在比赛限定环境下确保系统真实可运行，而不是只在高配开发机上成立。
- 建议内容：
  - 固定 `LoongArch + 银河麒麟` 的兼容性说明和安装路径
  - 固定 `4 核 CPU + 8GB 内存 + 256GB 硬盘` 下的最小运行方案
  - 把 `云端模型 API + 流式返回` 定为主链路
  - 把 `PostgreSQL + 全文检索 + 轻量 rerank` 定为主检索底座
  - 将 `Milvus / Neo4j / 本地 7B 大模型` 明确排除出当前必选栈
- 业务价值：
  - 避免方案脱离比赛真实环境
  - 提高部署成功率和答辩可信度

### P1：应该继续做

#### 4.7 案例审核 Patch 化与知识版本化

- 目标：把案例审核从“通过/驳回”升级成“结构化知识补丁审查”。
- 建议内容：
  - 审核页展示：
    - 原知识片段
    - 新建议内容
    - 修改原因
    - 证据来源
  - 审核通过后不只是写入新文档，还要保留 patch 记录和版本关系
  - 后台保留“原始生成内容”和“人工最终修正内容”的差异
- 业务价值：
  - 更贴近外部文档里强调的 diff patching / human-in-the-loop
  - 为后续模型优化积累高质量修正数据

#### 4.8 统一模型网关与降级策略

- 目标：把当前的 provider 切换提升为正式模型路由层。
- 建议内容：
  - 抽象统一模型网关，屏蔽云端模型、本地模型、多模态模型差异
  - 增加：
    - 超时控制
    - 重试
    - 熔断
    - 云端优先 / 本地轻量兜底
    - 成本与耗时日志
  - 对不同任务场景进行模型分配，例如：
    - OCR / 图片理解
    - 复杂诊断推理
    - 低风险结构化总结
- 业务价值：
  - 更符合“本地部署或云端服务均可运行”的赛题叙事
  - 也更利于后续适配国产算力环境

#### 4.9 真正的多模态统一检索

- 目标：把当前“文本 + OCR + 图片分析辅助”继续升级为统一向量空间检索。
- 建议内容：
  - 引入图文统一 embedding 路径
  - 图片、OCR 文本、设备型号、故障描述共同构造候选集
  - 保留当前稀疏检索和 PostgreSQL 全文检索作为精确锚定层
  - 在比赛阶段避免以重型向量库作为首选底座，优先把语义路径做通
- 业务价值：
  - 让“图片检索”从演示加分项变成稳定能力
  - 贴合外部文档所强调的多模态检索架构

### P2：可以在时间充足时做

#### 4.10 基于人工修正数据的偏好优化

- 目标：把审核差异沉成模型优化数据。
- 建议内容：
  - 记录初始建议与人工最终修正之间的差异
  - 形成内部偏好样本，为后续 prompt 优化或微调使用
- 业务价值：
  - 强化“系统会持续进化”的叙事
  - 但前提是 patch、版本和修正数据已经足够稳定

#### 4.11 面向本地部署的性能优化专题

- 目标：为后续龙芯、本地推理、边缘部署口径预留技术空间。
- 建议内容：
  - 增加本地模型运行基线
  - 补离线模式说明
  - 记录不同模型、不同部署形态下的延迟与资源占用
- 业务价值：
  - 对比赛题意和国产化叙事有帮助
  - 但不应在当前阶段挤占主链路完善时间

## 5. 现阶段不建议优先做

### P0：当前明确不建议先做

#### 5.1 不建议先拆微服务

- 原因：
  - 当前系统仍处于比赛和演示收口阶段
  - 单体 FastAPI 已足够支撑现有链路
  - 提前拆服务只会增加部署、调用、日志和排障复杂度

#### 5.2 不建议先引入完整图数据库平台

- 原因：
  - 当前真正缺的是补丁化审核、版本化知识、证据链
  - 在 patch、版本和关系建模尚不稳定时，上 Neo4j 一类平台收益偏低

#### 5.3 不建议先做 RLHF / DPO 微调

- 原因：
  - 现在还缺稳定、结构化、可复用的人工修正数据
  - 先做微调容易把问题掩盖在模型层，而不是解决业务结构问题

#### 5.4 不建议先做设备控制闭环自动执行

- 原因：
  - 工业检修场景风险高
  - 当前系统更适合定位为“辅助决策 + 合规校验 + 人工授权”的平台
  - 直接迈向自动控制会显著增加安全风险与答辩质疑点

### P1：可以延后，不必现在投入

#### 5.5 不建议先上复杂分布式任务平台

- 原因：
  - 当前异步导入任务量有限
  - 先用现有最小 worker 验证流程和状态机更务实
  - 等到真实并发成为瓶颈，再考虑 Celery、RQ 等方案

#### 5.6 不建议先投入重型向量基础设施

- 原因：
  - 当前 PostgreSQL 全文检索、规则召回、rerank 还有继续提升空间
  - 先把数据质量、锚点结构和评测集补齐，收益通常大于盲目换库
  - 在 `4 核 CPU + 8GB 内存` 环境下，重型平台还会额外放大资源与部署压力

#### 5.7 不建议先把本地大模型部署做成主验收目标

- 原因：
  - 当前比赛硬件资源过于受限
  - 主链路真正更重要的是“云端稳定调用 + 流式响应 + 可降级”
  - 过早追逐本地大模型完整部署，会直接挤占主链路和交付物收口时间

## 6. 推荐执行顺序

### 第一阶段

- 国产化运行环境说明与轻量化选型固化
- 统一模型网关
- 硬状态机与证据门禁
- 正式主链路事件流化

### 第二阶段

- 轻量多模态 RAG
- 层级化知识锚点与来源定位深化
- 评测集与业务指标固化

### 第三阶段

- 案例审核 Patch 化与知识版本化
- 多模态统一检索
- 本地部署性能优化专题

## 7. 一句话结论

项目下一阶段的正确方向，不是继续增加“更多页面”或“更多模型名词”，而是把当前系统补成一个具备 `轻量化可运行、模型可切换、步骤可门禁、知识可溯源、交付可验收` 的检修作业平台。
