## 1. 文档目的

本文档用于明确「工业多模态检修系统」的首期业务定位、试点范围、核心痛点、业务边界、目标价值与落地约束，为后续《用户角色与使用场景文档》《检修业务流程与 SOP 文档》《MVP 产品需求文档》提供统一的业务依据。

本文档重点回答以下问题：

1. 本系统首期服务谁；
2. 本系统首期聚焦什么设备与什么场景；
3. 本系统首版解决哪一段检修流程问题；
4. 本系统首版不解决什么问题；
5. 如何判断该系统具备初步落地价值。

---

## 2. 业务背景

在工业设备检修场景中，一线人员长期面临「故障发现容易、资料定位困难、经验传递缓慢、现场指导不足」的问题。传统检修方式高度依赖纸质手册、老师傅经验和电话沟通，存在以下共性不足：

- 检修知识分散在 PDF、纸质 SOP、历史工单和个人经验中，现场检索效率低；
- 新人面对异常现象时，难以快速判断故障点和下一步操作；
- 检修过程记录不完整，难以形成高质量案例沉淀；
- 在网络受限、数据敏感或国产化要求较高的工业现场，通用云端智能方案难以直接应用。

基于上述背景，本项目拟建设一套面向工业现场的多模态检修辅助系统，通过图像、语音和文本等输入方式，帮助一线检修人员快速获取与当前设备和故障现象相关的 SOP、历史案例和处理建议，并将一次检修过程沉淀为可复用的知识资产。主线同时覆盖行业常见的知识断层、多模态信息割裂、实时决策难、本地化部署与审计溯源等诉求。

**对标说明（便于评审对照）：** 业务表述与第十五届中国软件杯大赛 **A1「基于多模态大模型技术的设备检修知识检索与作业系统」**（出题单位：龙芯中科）在「多模态检索 + 标准化作业引导 + 知识积累 + 国产化环境」方向上保持一致；首期试点在设备类别上聚焦机泵，知识库可包含赛题提供的参考手册材料与自选资料，见第 3.6 节与第 9 节。

## 3. 首期试点范围

### 3.1 首期试点对象

本项目首期试点聚焦于**关键机泵类设备**的辅助检修场景，优先覆盖泵体、法兰、密封件、轴承等相关部件的常见故障排查与 SOP 指导。

### 3.2 首期试点故障范围

首版系统优先支持以下高频、易描述、易采集的故障现象：

- 法兰渗漏；
- 异响；
- 振动异常；
- 温度异常；
- 密封件老化或损坏；
- 常规拆检前的 SOP 核验。

### 3.3 首期试点流程范围

本项目首版仅聚焦以下主流程：

**现场发现异常 → 上传图像/语音/文字描述 → 系统检索相关 SOP 与案例 → 输出辅助检修建议 → 标准化分步作业引导（含合规提示）→ 记录处理结果 → 沉淀知识条目**

### 3.4 首期试点部署范围

首版系统优先验证以下部署形态：

- **交付形态**：采用 **B/S（浏览器/服务器）架构**，客户端为 **PC Web 浏览器**与/或**移动终端浏览器**（或 App 内嵌 WebView），业务逻辑与模型服务部署在服务端，以满足赛题对 B/S 与可视交互的要求。
- **目标验收/竞赛环境**：正式评测与现场答辩时，**服务端须在龙芯 LoongArch 架构、银河麒麟高级服务器操作系统（V10/V11）上部署运行**（与赛题及龙芯云平台环境一致；赛题明确不满足该运行环境要求时功能项不得分，本系统将其列为硬验收约束）。
- **开发期环境**：允许在 x86 开发与联调环境完成功能迭代，但需保留与 LoongArch + 麒麟的兼容性验证计划（容器/依赖架构、字节序与依赖库等），并在部署文档中说明差异与验证结果。

### 3.5 首期不纳入范围

为保证项目首版可实现、可演示、可验收，以下内容不纳入本期范围：

- 全厂所有设备类型统一接入；
- PLC 直接控制、设备自动启停等闭环控制；
- 预测性维护模型的完整落地；
- 数字孪生平台联动；
- ERP、采购、财务等经营管理流程；
- 全自动无人审批的高危作业执行。

### 3.6 与 A1 赛题基本功能的对齐说明

| 赛题要点 | 本系统首版业务表述 |
| -------- | ------------------ |
| 多模态知识检索（文本、故障图像、设备型号等） | 支持文本、语音/录音、现场照片；**设备型号/设备编号/台账关键字**与现象联合检索；检索侧同时支持 **语义检索与关键词/模糊匹配**（如手册章节号、部件俗称近似匹配）。 |
| 标准化作业引导（全流程、分步、合规、按设备类型与检修等级个性化） | 在《检修业务流程与 SOP 文档》中定义 **检修等级 / 流程模板**（如日常保养、定修、抢修），按设备类型与所选等级挂载 SOP 节点序列，逐步展示操作与安全提示。 |
| 大模型本地或云端接入 | 支持本地部署或 API 方式接入大模型服务（如通义、DeepSeek 等），具体以《部署与运维文档》为准；业务上坚持「先检索后生成」。 |
| 知识积累与更新 + 人工标注纠正 LLM 输出 | 支持案例与经验经审核入库；支持专家对 **单次模型输出** 进行 **纠错、正误标注与批注**，并可按需 **导出标注样本** 供 RAG 评测、提示词优化或后续模型微调（首版以实现闭环与留痕为主，不强制完整 MLOps）。 |
| 官方参考数据 | 可将赛题提供的 **《摩托车发动机维修手册》** 等电子手册经解析后纳入知识库，用于验证检索与作业引导链路；**现场试点叙事仍以机泵为主**，二者在文档中并列说明来源与用途，避免评委误解为仅支持摩托车。 |

## 4. 核心业务痛点

围绕首期试点范围，当前业务痛点归纳如下。

| 痛点维度       | 当前现状                                   | 首版系统拟解决的问题                           |
| -------------- | ------------------------------------------ | ---------------------------------------------- |
| 故障定位慢     | 一线人员依赖经验判断或翻阅手册，定位耗时长 | 快速根据现象与设备上下文匹配相关 SOP、案例和排查建议 |
| 资料检索难     | 手册和记录分散，现场很难精准找到所需内容   | 通过图像、语音、设备型号/关键词进行语义与模糊相结合的检索 |
| 经验传承弱     | 老员工经验隐性化，新人学习成本高           | 将典型工单与处理过程沉淀为结构化知识           |
| 现场指导不及时 | 复杂故障需多轮沟通，现场等待时间长         | 提供首轮辅助建议，并支持升级为专家会诊流程     |
| 检修闭环不完整 | 处理过程和结果记录松散，后续复盘困难       | 将异常、建议、操作、结果与引用来源统一归档     |
| 安全审计要求高 | 高危操作不能仅依赖经验或 AI 直接决策       | 高危步骤必须带出处、带审批、带可追溯记录       |
| 部署环境受限制 | 一些工业现场网络受限、数据不宜出厂         | 支持本地化部署与离线条件下的核心能力保底运行   |
| 作业非标难监管 | 口头传授多、工步与合规提醒不统一             | 嵌入分步作业与合规提醒，并按检修等级选择流程模板 |

## 5. 目标业务场景

本系统首版围绕一个主场景和若干扩展场景展开。

### 5.1 主场景：现场辅助检修 + 标准化作业引导

当一线检修人员在机泵设备现场发现渗漏、异响、振动异常等问题时，可通过移动端拍照、录音或语音提问的方式发起检修辅助请求。系统根据**设备型号/台账**、故障现象和知识库内容，进行 **语义检索与模糊匹配**，返回相关 SOP 条目、历史案例和建议步骤；在工人确认检修等级（或由默认值）后，按流程模板 **分步展示工步、工具备件与安全合规提示**，帮助现场人员缩短查找资料与规范作业的时间。

### 5.2 扩展场景 A：疑难故障升级会诊

当系统未命中高置信度结果，或故障属于高风险、低频、复杂异常时，现场人员可发起升级会诊。系统将异常信息、历史台账和多模态证据整合为会诊工单，推送给检修班长或技术专家进行复核与审批。

### 5.3 扩展场景 B：本地化与国产化部署

在网络受限、数据敏感或有国产化要求的工业环境中，系统支持采用本地化部署方式运行核心检索与辅助决策能力，确保检修数据、现场图片和作业记录在局域网内闭环处理；正式验收以 **LoongArch + 银河麒麟服务器版** 为准。

### 5.4 扩展场景 C：模型输出人工纠错与标注回流

专家对系统生成的辅助文本可进行 **错误纠正、正误判断与批注**；经授权后，标注结果可作为 **知识库补丁候选** 或 **导出为评测/微调样本**，用于持续改进检索排序、提示词或模型表现（首版以流程与数据留痕为主）。

## 6. 业务边界

### 6.1 首版纳入范围

首版系统纳入以下业务能力：

- 基于图像、语音、文本的异常信息采集；
- **设备型号/编号/台账关键字**与现象结合的 SOP 与历史案例检索；
- **语义检索 + 模糊/关键词匹配** 相结合的检索策略；
- 对检修建议附带出处引用与安全提示；
- **按设备类型与检修等级挂载的标准化分步作业引导与合规提醒**；
- 对低置信度、高风险场景发起人工升级；
- 对检修过程与结果进行结构化记录；
- 将已完成工单沉淀为可复用知识条目；
- **专家对 LLM 输出的人工纠错与标注**，及可选的样本导出；
- **B/S 架构**下的 PC Web 与移动端访问能力；
- 在 **LoongArch + 银河麒麟高级服务器操作系统** 上的部署与运行验证（与赛题硬约束对齐）。

### 6.2 首版不纳入范围

首版系统不直接承担以下职责：

- 代替人工做最终安全责任判定；
- 自动控制设备执行启停、拆解或参数下发；
- 替代企业正式 EAM/MES/ERP 系统；
- 对所有未知故障给出无依据的自由生成答案；
- 在未经审批的情况下输出高危操作指令。

### 6.3 业务原则

本系统在业务上遵循以下原则：

- **辅助决策，不替代责任主体**；
- **先检索后生成，先依据后建议**；
- **高危必审批，建议可追溯**；
- **数据尽量本地闭环，记录全程可审计**。

## 7. 预期业务价值

### 7.1 对一线检修人员的价值

- 缩短现场查找资料和确认步骤的时间；
- 降低因经验不足导致的漏检、误判和误操作风险；
- 让现场指导更加直接，减少「只给原理、不告诉下一步怎么做」的情况；
- 在统一界面内完成「查资料—跟工步—留记录」。

### 7.2 对技术专家与班组管理的价值

- 将重复咨询和重复解释转化为可复用知识；
- 通过标准化记录提升工单复盘效率；
- 对高危作业形成可追溯的审批链和责任链；
- 通过纠错与标注积累可治理的模型反馈数据。

### 7.3 对企业管理层的价值

- 沉淀隐性检修经验，提升知识资产化水平；
- 为后续知识库扩展、设备台账治理和预测性维护提供数据基础；
- 在符合本地化和国产化要求的前提下，探索智能检修辅助的应用路径。

## 8. 量化评价指标与验收口径

为保证项目首版可测、可比、可验收，定义如下核心指标。

| 指标名称     | 指标定义                                                     | 目标值 | 测量方式                                       | 对照基线              |
| ------------ | ------------------------------------------------------------ | ------ | ---------------------------------------------- | --------------------- |
| 检索命中率   | 对给定故障问题，系统返回结果中正确 SOP/案例位于 Top-1 的比例 | ≥ 80%  | 基于人工标注的测试集统计                       | 人工关键词查找        |
| 响应时间     | 从用户提交问题到系统返回首条可用建议的总耗时                 | ≤ 3 秒 | 在指定演示环境下多次采样取平均值               | 人工查阅纸质/PDF 手册 |
| 来源可追溯率 | 系统给出的建议中带有明确出处的比例                           | = 100% | 检查输出结果是否包含来源文档、章节或节点信息   | 传统口头经验指导      |
| 处理效率提升 | 使用系统后完成首轮排查所需时间的缩减比例                     | ≥ 20%  | 对比「人工查手册流程」与「系统辅助流程」的平均耗时 | 人工查找流程          |
| 知识沉淀率   | 已完成工单中能够转化为结构化知识条目的比例                   | ≥ 60%  | 统计可用工单中成功进入知识库的数量             | 无统一沉淀机制        |
| 升级准确性   | 在低置信度或高风险条件下，系统正确触发人工升级的比例         | ≥ 90%  | 对预设异常样本进行规则校验                     | 纯人工经验判断        |

说明：

1. 上述目标值用于首版演示与中期验收，可根据真实测试结果进行调整；
2. **在 LoongArch + 银河麒麟目标环境**下，响应时间指标应单独记录（允许略高于 x86 开发环境，但须在测试报告中注明硬件与操作系统版本）；
3. 高危场景不以「自动执行成功率」为目标，而以「正确阻断与正确升级率」为目标；
4. 「≤3 秒」默认指**检索与首轮结构化建议**返回时间；若包含大模型长文本生成，可在测试报告中拆分为「首屏可用建议」与「完整报告」两档分别统计。

## 9. 关键假设与落地前提

本项目首版落地建立在以下前提下：

1. 试点企业或模拟环境能够提供一定数量的设备台账、SOP 文档和历史检修记录；
2. 现场人员具备基础移动终端使用能力，能够完成拍照、录音和语音提问；
3. 部署环境具备基本的局域网或本地服务运行条件；
4. 高危操作场景中，企业仍保留人工审批和安全责任机制；
5. 知识库首版允许通过人工清洗和半结构化整理方式完成冷启动；
6. **竞赛/验收环境**：可获得或可迁移至 **龙芯云平台** 所提供或等效的 **LoongArch + 银河麒麟高级服务器版** 环境，并完成部署验证；
7. **赛题参考数据**：可获取并解析赛题提供的参考手册（如《摩托车发动机维修手册》电子版），用于检索与作业引导演示；机泵类自有资料与之并行维护，不互相替代业务定位。

## 10. 风险与应对策略

| 风险类别     | 风险描述                                 | 应对策略                                                     |
| ------------ | ---------------------------------------- | ------------------------------------------------------------ |
| 业务范围失控 | 项目试图同时覆盖过多设备和过多能力       | 首版仅聚焦关键机泵设备与高频故障，其他能力延期               |
| 数据准备不足 | 设备资料不完整，知识库冷启动效果有限     | 优先整理高频故障 SOP、典型案例和核心设备台账                 |
| 模型误答风险 | 系统输出与知识库不一致的建议             | 强制采用「先检索后生成」与规则拦截机制，未命中则明确提示无结果 |
| 安全责任风险 | 用户误将系统建议视为最终操作指令         | 高危步骤强制审批，输出中固定标注「辅助建议」与来源依据         |
| 现场环境干扰 | 图片模糊、语音嘈杂、网络中断影响使用体验 | 提供重试提示、离线 SOP 保底模式和异常回退机制                |
| 部署性能不足 | 本地或国产化环境下推理或检索延迟较高     | 首版优先保证核心链路可用；检索侧可缓存、异步生成；测试报告分环境披露 |
| 知识老化风险 | SOP 或案例长期不更新导致建议失真         | 建立报错、审核、版本更新和知识回收机制                       |
| 目标平台适配 | LoongArch + 麒麟下依赖不兼容或性能不达标 | 尽早纳入 CI/手工矩阵验证；依赖选型优先选有国产化生态的组件   |

---

## 11. 数据安全与合规要求

考虑到工业检修场景涉及设备台账、运行日志、图像资料与作业记录，系统需满足以下要求：

- **数据本地优先**：优先采用本地或局域网部署方式处理知识检索与作业数据；
- **权限分级控制**：不同角色仅访问其职责范围内的设备资料和工单信息；
- **建议全程溯源**：所有输出建议必须保留来源文档或知识节点引用；
- **操作过程留痕**：关键诊断、审批、知识更新、**模型输出纠错与标注**行为需记录日志；
- **敏感信息脱敏**：涉及人员隐私、内部编码和敏感工艺参数的数据需脱敏展示。

## 12. 方案特点与差异化方向

首版系统的差异化不在于「替代人」，而在于「把检修现场最痛的几个环节串起来」。其主要特点如下：

### 12.1 多模态输入更贴近现场

系统支持图像、语音和文本作为输入入口，更符合工业现场「戴手套、不方便长文本输入、需要快速上手」的使用特征。

### 12.2 检索增强而非自由发挥

系统以设备台账、SOP 和历史工单为核心依据，综合 **语义向量检索与关键词/模糊匹配**，优先返回有来源的检索结果，再在此基础上组织建议内容，降低无依据输出风险。

### 12.3 安全与审批前置

首版从业务上明确高危操作不可绕过人工审批，系统定位为「辅助检修」而非「自动执行」。

### 12.4 兼顾国产化与 B/S 交付

系统采用 **B/S 架构**，并将在 **LoongArch + 银河麒麟服务器操作系统** 上完成运行验证，与赛题及出题方生态对齐。

### 12.5 知识闭环与模型反馈闭环

除案例审核入库外，支持 **对 LLM 输出的人工标注与纠错回流**，使知识库与模型侧均可迭代改进。

---

## 13. 后续演进方向

在首版完成「辅助检修 + 知识闭环」后，系统可按以下方向逐步演进：

1. 从单类设备扩展到更多设备类型；
2. 从故障后辅助排查扩展到结合时序数据的预警分析；
3. 从单点检修辅助扩展到与数字孪生、设备管理平台的联动；
4. 从规则与检索主导，逐步扩展到更复杂的多智能体协同诊断；
5. 将标注样本与线上日志接入更系统的 RAG 评测与模型迭代流水线。

## 14. 术语说明

| 术语 / 缩写 | 含义                                                         |
| ----------- | ------------------------------------------------------------ |
| SOP         | 标准作业程序，用于指导设备检修步骤与安全要求                 |
| RAG         | 检索增强生成，指系统先检索知识库内容，再生成辅助建议         |
| MTTR        | 平均修复时间，用于衡量设备故障从发现到恢复的平均耗时         |
| LoongArch   | 龙芯自主指令系统架构；本赛题要求服务端运行于该架构           |
| 银河麒麟    | 银河麒麟高级服务器操作系统（V10/V11），本赛题指定服务端 OS   |
| B/S         | 浏览器/服务器架构：客户端以浏览器为主，核心逻辑在服务端      |
| 多模态      | 同时处理图像、语音、文本等多种输入形式                       |
| 检修等级    | 如日常保养、计划定修、抢修等，用于选择不同的标准作业流程模板 |
