## 1. 文档目的

本文档用于明确“工业多模态检修系统”在首期试点阶段的用户角色、职责边界、核心使用场景与交互方式，为后续的产品需求、原型设计、权限设计、接口设计和测试验收提供统一依据。

本文档重点回答以下问题：

1. 系统首期由哪些角色使用；
2. 各角色分别在检修流程中承担什么职责；
3. 各角色与系统之间如何交互；
4. 系统首版重点支撑哪些典型业务场景；
5. 在异常、断网、高危操作等情况下如何降级与流转。

------

## 2. 适用范围

本文档适用于首期试点范围内的机泵类设备辅助检修业务，重点覆盖以下场景：

- 一线人员发现设备异常后的现场辅助排查；
- 常规问题未解决时的升级会诊；
- 高危步骤前的审批确认；
- 检修完成后的工单归档与知识沉淀。

本文档不涵盖以下内容：

- 全厂全设备类型统一角色模型；
- 自动控制设备启停或参数下发；
- ERP、采购、财务等经营管理角色；
- 预测性维护、数字孪生等远期扩展角色体系。

### 2.1 交付形态与目标运行环境（与 A1 赛题一致）

- **架构**：系统以 **B/S（浏览器/服务器）** 交付，现场与办公室通过浏览器访问同一套服务端；移动端可采用浏览器或 App 内嵌 Web 方式接入，业务逻辑不落在纯离线单机 exe（与赛题「PC Web 或 App 可视界面」一致）。
- **目标验收/竞赛运行环境**：服务端须在 **龙芯 LoongArch + 银河麒麟高级服务器操作系统（V10/V11）** 上部署运行；开发期可在 x86 环境联调，但文档与测试报告需区分环境并记录目标平台上的验证结论。
- **检索与输入**：除多模态外，支持 **设备型号/编号/台账关键字** 与现象联合检索；检索能力在业务上要求 **语义检索与模糊/关键词匹配** 相结合（详见《业务场景定义文档》第 3.6 节）。

## 3. 角色体系定义

首期版本共定义 4 类核心角色。

### 3.1 一线检修工人

**角色定位：**
 系统的主要使用者，负责在设备现场发现异常、发起问题、执行标准检修动作并回填处理结果。

**典型工作环境：**
 高噪音、光线复杂、可能有油污和粉尘，存在佩戴手套、头盔、护具等情况。

**常用终端：**
 手机、工业平板、便携式移动终端。

**核心诉求：**

- 快速判断当前异常可能对应什么问题；
- 快速找到适用于当前设备和现象的 SOP；
- 获得清晰、简洁、可执行的下一步建议；
- 尽量减少打字和复杂页面操作；
- 在必要时快速发起人工升级求助。

**职责边界：**

- 可以发起检修辅助请求；
- 可以查看系统返回的建议与出处；
- 可以执行经授权的检修步骤；
- 可以提交现场图片、语音、处理结果；
- 不负责修改正式 SOP；
- 不具备高危步骤最终审批权限。

------

### 3.2 检修班长 / 技术专家

**角色定位：**
 负责对复杂问题进行技术复核、对升级工单进行会诊支持，并对知识条目质量进行专业把关。

**典型工作环境：**
 办公室、集控室、检修值班室，必要时到现场支援。

**常用终端：**
 PC Web 端、管理后台、必要时使用移动端查看现场信息。

**核心诉求：**

- 快速了解现场异常与系统初步判断；
- 对复杂问题进行技术复核与补充；
- 审核系统建议是否符合现场实际；
- 将典型案例沉淀为标准知识；
- 对班组整体处理情况进行追踪；
- 对系统或大模型生成的辅助文本进行 **纠错、正误标注与批注**，支撑知识库修订与后续模型/RAG 优化。

**职责边界：**

- 可查看升级工单和现场多模态证据；
- 可审核、修订、补充系统生成建议；
- 可对单次会话中的模型输出进行 **人工纠错与标注**（在授权范围内）；
- 可批准或驳回非高危复杂操作建议；
- 可发起知识沉淀申请；
- 不负责系统底层运行维护。

------

### 3.3 安全审批人

**角色定位：**
 对涉及断电、拆卸、挂牌上锁、可能带来人身或设备风险的关键步骤进行安全确认与授权。

**典型工作环境：**
 现场安全管理岗位、班组安全责任岗位或值班审批岗位。

**常用终端：**
 PC Web 端、审批端、移动审批页面。

**核心诉求：**

- 明确看到拟执行操作的风险等级；
- 查看关键安全凭证是否齐全；
- 确保高危操作在授权前不被执行；
- 保留审批留痕和责任链。

**职责边界：**

- 可查看高危工单及相关安全凭证；
- 可对高危步骤进行批准、驳回或要求补充材料；
- 不直接修改业务知识库；
- 不承担系统运维职责。

------

### 3.4 系统管理员 / 知识管理员

**角色定位：**
 负责系统运行维护、权限配置、知识库更新、版本管理与日志审计。

**典型工作环境：**
 IT 运维环境、后台管理端、本地服务器管理环境。

**常用终端：**
 PC 管理后台、运维控制台。

**核心诉求：**

- 保障系统稳定运行；
- 管理用户与角色权限；
- 管理知识条目、版本和发布状态；
- 跟踪日志、告警和异常状态；
- 支持本地化部署与更新维护；
- 在启用反馈闭环时，管理 **标注样本导出权限**、脱敏规则与保留周期。

**职责边界：**

- 可配置系统、角色、权限和知识发布流程；
- 可管理知识条目的上架、回滚和版本；
- 可配置是否允许导出纠错/标注数据集及审批流程（若首版启用）；
- 可查看系统运行状态和审计日志；
- 不替代技术专家进行专业诊断；
- 不替代审批人进行安全授权。

------

## 4. 角色关系说明

在首期业务闭环中，四类角色之间形成如下协作关系：

- 一线检修工人负责发现异常、发起请求、执行动作、提交结果；
- 检修班长 / 技术专家负责技术复核、升级会诊和知识把关；
- 安全审批人负责高危操作前的安全授权；
- 系统管理员 / 知识管理员负责系统与知识资产的持续运维。

系统本身定位为辅助平台，不替代任何角色的正式责任主体。

## 6. 典型使用场景总览

首期版本重点支撑以下 7 个典型场景：

1. 现场异常快速提问与 SOP 检索；
2. 常规故障首轮辅助排查；
3. 未命中或复杂故障升级会诊；
4. 高危步骤前安全审批；
5. 检修完成后的知识沉淀；
6. **按设备类型与检修等级的标准化分步作业引导**（含合规提示）；
7. **模型输出人工纠错与标注回流**（可选启用导出，用于评测或后续优化）。

其中，场景 1、2、6 为首版高频主场景；场景 3、4、5、7 为保证业务闭环及赛题能力覆盖的关键支撑场景。

## 7. 详细使用场景定义

------

### UC-01 现场异常快速提问

**场景名称：**
 现场异常快速提问

**参与角色：**
 一线检修工人

**场景目标：**
 让工人在发现机泵异常后，能够在最短时间内获得与当前设备和现象相关的参考信息。

**触发条件：**

- 工人在现场发现异常现象；
- 具备拍照、录音或语音输入条件；
- 已进入对应设备页面或已识别当前设备。

**输入：**

- 现场照片；
- 语音问题或文字问题；
- 可选的设备型号、设备编号、位置、部件信息。

**主流程：**

1. 工人在移动端进入设备问题提问入口；
2. 上传现场照片，并通过语音描述异常现象；
3. 系统识别设备信息、异常关键词和问题意图；
4. 系统检索匹配的 SOP、案例和知识条目；
5. 系统返回简洁建议、相关步骤和出处；
6. 工人查看结果，决定继续执行、重新提问或升级会诊。

**输出：**

- 问题识别结果；
- 候选 SOP / 案例；
- 首轮排查建议；
- 引用来源信息。

**成功标准：**

- 工人能在较少操作下拿到可参考的首轮指导；
- 结果带有明确出处；
- 无命中时系统明确提示并引导下一步。

------

### UC-02 常规故障首轮辅助排查

**场景名称：**
 常规故障首轮辅助排查

**参与角色：**
 一线检修工人

**场景目标：**
 对于常见机泵故障，系统帮助工人完成从“看到异常”到“按步骤排查”的首轮闭环。

**适用故障类型：**

- 法兰渗漏；
- 异响；
- 振动异常；
- 温度异常；
- 密封件可疑老化。

**主流程：**

1. 工人发起问题；
2. 系统返回相关 SOP 节点和排查建议；
3. 工人根据提示执行基础检查；
4. 工人回填检查结果；
5. 系统根据结果更新下一步建议或引导升级。

**输出：**

- 分步排查建议；
- 安全提示；
- 需补充的检查项；
- 处理结果记录。

**注意事项：**

- 系统建议必须基于已有知识；
- 对涉及高危动作的步骤不直接放行；
- 若置信度不足，自动转入升级路径。

------

### UC-03 疑难故障升级会诊

**场景名称：**
 疑难故障升级会诊

**参与角色：**
 一线检修工人、检修班长 / 技术专家

**场景目标：**
 在常规检索无效、故障复杂或现场风险较高时，将问题升级给技术专家进行复核。

**触发条件：**

- 系统未命中高置信度结果；
- 工人手动点击“升级会诊”；
- 问题被识别为复杂异常；
- 当前建议包含多种可能根因，无法直接执行。

**输入：**

- 现场照片、语音、视频等多模态证据；
- 系统初步检索结果；
- 工人补充描述；
- 相关设备历史记录。

**主流程：**

1. 工人发起升级会诊；
2. 系统自动整理问题摘要、设备信息和初步检索结果；
3. 检修班长 / 技术专家接收升级工单；
4. 专家查看现场证据、系统建议和相关资料；
5. 专家补充、修订或确认建议；
6. 系统将会诊结论返回给现场工人。

**输出：**

- 升级工单；
- 技术复核意见；
- 修订后的处理建议；
- 后续执行要求。

**成功标准：**

- 升级信息结构清晰，不需要工人反复重复问题；
- 专家能快速定位上下文；
- 现场人员获得明确结论或后续动作要求。

------

### UC-04 高危步骤审批

**场景名称：**
 高危步骤审批

**参与角色：**
 一线检修工人、安全审批人、检修班长 / 技术专家

**场景目标：**
 保证涉及高危操作的步骤在执行前经过正式授权。

**典型适用场景：**

- 断电操作；
- 挂牌上锁；
- 设备拆卸；
- 涉及安全防护确认的步骤。

**触发条件：**

- 系统识别当前步骤为高危；
- 技术专家建议中包含高危动作；
- 工单风险等级达到审批阈值。

**输入：**

- 待执行步骤；
- 风险等级；
- 现场安全凭证；
- 相关 SOP 和来源依据。

**主流程：**

1. 系统拦截高危步骤；
2. 工人补充安全凭证；
3. 安全审批人查看材料并决定批准、驳回或要求补充；
4. 审批结果同步给现场人员；
5. 批准后工人方可继续执行后续流程。

**输出：**

- 审批结论；
- 审批意见；
- 审批时间与责任留痕。

**业务原则：**

- 未审批不得执行；
- 审批动作必须留痕；
- 审批结论必须与具体步骤绑定；
- 审批通过不等于免除现场责任。

------

### UC-05 检修完成与知识沉淀

**场景名称：**
 检修完成与知识沉淀

**参与角色：**
 一线检修工人、检修班长 / 技术专家、系统管理员 / 知识管理员

**场景目标：**
 将已完成的检修记录转化为可复用知识，持续提升知识库质量。

**触发条件：**

- 工单已完成；
- 现场结果和处理过程已上传；
- 存在可复用的经验或新案例。

**输入：**

- 工单处理记录；
- 现场多模态材料；
- 专家补充说明；
- 原始建议与最终结果。

**主流程：**

1. 工人提交处理结果；
2. 技术专家确认处理过程是否具备沉淀价值；
3. 系统整理为候选知识条目；
4. 知识管理员进行格式审核、版本管理和发布；
5. 新条目进入知识库供后续检索使用。

**输出：**

- 可审核知识条目；
- 知识版本记录；
- 发布状态。

**成功标准：**

- 能从真实工单中沉淀出结构化案例；
- 知识条目有明确来源、审核人和版本；
- 不把未经确认的信息直接写入正式知识库。

------

### UC-06 标准化分步作业引导

**场景名称：**  
按设备类型与检修等级的分步作业引导

**参与角色：**  
一线检修工人（执行）；检修班长 / 技术专家（维护流程模板）；系统管理员 / 知识管理员（发布模板版本）

**场景目标：**  
将「检索到的 SOP」落实为可执行的 **分步工单式引导**：按所选 **检修等级**（如日常保养、计划定修、抢修）展示工步序列、工具备件、合规与安全提示，满足赛题对「标准化作业引导、按设备类型与检修等级个性化」的要求。

**触发条件：**

- 已完成设备上下文确认（或默认设备）；
- 系统已返回至少一条可用 SOP 或流程模板；
- 工人选择或与默认一致的检修等级。

**主流程：**

1. 工人确认设备类型与检修等级（或使用班组默认等级）；
2. 系统加载对应 **流程模板**，按序展示工步；
3. 每步展示操作说明、合规提醒、是否高危/需审批；
4. 工人逐步确认已完成项或跳过（受权限与规则约束）；
5. 遇高危未审批工步时 **阻断** 并跳转 UC-04；
6. 工步完成后进入结果回填或结单流程。

**输出：**

- 工步执行与确认记录；
- 合规与安全提示留痕；
- 与工单状态机关联的进度状态。

**成功标准：**

- 工人能按界面顺序完成「该等级下」应做的标准动作；
- 高危步骤不会被未授权绕过；
- 记录可追溯，便于复盘与知识沉淀。

### UC-07 模型输出纠错与标注回流

**场景名称：**  
模型输出人工纠错与标注回流

**参与角色：**  
检修班长 / 技术专家；系统管理员 / 知识管理员（导出与策略）

**场景目标：**  
对单次大模型辅助输出进行 **事实性与合规性纠错**，打 **正/误标签** 或批注，可选 **生成候选知识修正** 或 **导出标注样本**，支撑赛题要求的「人工标注和纠正 LLM 输出以改进系统」。

**触发条件：**

- 专家在会诊或复核中认为模型某段输出存在错误、遗漏或易误导；
- 或质检/复盘流程中主动发起标注。

**主流程：**

1. 专家在 Web 端打开对应会话或工单中的模型输出片段；
2. 选择纠错类型（事实错误、引用不当、安全提示缺失等），填写修正说明；
3. 可选：将修正合并为知识库补丁候选，走既有审核发布流程；
4. 若启用导出：由知识管理员按策略导出脱敏后的 JSON/表格供评测或提示词迭代。

**输出：**

- 纠错与标注记录（绑定工单、消息 ID、操作者、时间）；
- 可选：候选知识修订单；
- 可选：导出的评测/微调样本包。

**成功标准：**

- 每条标注可追溯到原始模型输出与上下文；
- 未经审核的纠错不直接覆盖正式知识库；
- 导出符合权限与脱敏要求。

## 8. 交互触点与终端形态

### 8.1 移动端

**主要用户：** 一线检修工人
 **主要任务：** 异常上报、问题提问、查看建议、回填结果
 **设计要求：**

- 语音和拍照优先；
- 页面层级少；
- 核心按钮大；
- 在戴手套场景下可操作；
- 适合快速中断和恢复。

### 8.2 PC Web 端

**主要用户：** 检修班长 / 技术专家、安全审批人、系统管理员
 **主要任务：** 查看工单、技术复核、审批、知识管理、日志管理
 **设计要求：**

- 信息密度较高；
- 支持多窗口查看上下文；
- 支持审批、审核、版本对比；
- 支持日志和历史记录追踪。

### 8.3 后台管理端

**主要用户：** 系统管理员 / 知识管理员
 **主要任务：** 用户与权限配置、知识发布、回滚、系统监控
 **设计要求：**

- 强调操作留痕；
- 强调权限边界；
- 支持版本与日志查询；
- 支持告警与异常排查。

## 9. 异常场景与降级策略

### 9.1 现场网络中断

**处理策略：**

- 移动端进入离线保底模式；
- 暂停升级会诊和在线检索；
- 允许查看本地预加载的基础 SOP；
- 网络恢复后自动同步离线期间记录。

### 9.2 图片或语音质量差

**处理策略：**

- 系统提示重拍、重录或改用文字输入；
- 当识别质量低于阈值时，不输出高置信度建议；
- 引导用户补充设备编号、部件信息或现象关键词。

### 9.3 知识库未命中

**处理策略：**

- 明确提示“未找到匹配内容”；
- 不允许系统自由编造答案；
- 引导用户发起升级会诊或转人工处理。

### 9.4 高危步骤未审批

**处理策略：**

- 阻断执行；
- 固定显示审批状态；
- 不允许通过刷新或重进页面绕过审批节点。

### 9.5 知识条目疑似失效

**处理策略：**

- 标记为待复核状态；
- 暂停推荐；
- 通知技术专家或知识管理员审核更新。

## 10. 用户体验目标

为保证首版系统“能用、敢用、愿意用”，定义以下体验目标：

### 10.1 一线工人体验目标

- 从进入问题页面到完成一次问题提交，操作步骤尽量控制在少量点击内；
- 结果表达以“下一步做什么”为核心，而不是大段理论描述；
- 所有建议必须带出处；
- 高危动作必须明确警示。

### 10.2 专家体验目标

- 升级工单具备完整上下文，不需要重复追问基础信息；
- 能看到系统为什么给出当前建议；
- 能快速补充、修订和确认意见。

### 10.3 审批体验目标

- 能一眼看到当前工单是否高危；
- 能快速查看审批所需材料；
- 审批行为必须留痕且可追溯。

### 10.4 管理体验目标

- 知识条目发布和回滚过程清晰；
- 权限配置简单明确；
- 异常日志和运行告警可查。

## 11. 角色设计原则

本系统的角色与使用场景设计遵循以下原则：

1. **围绕首期试点，不做大而全角色体系；**
2. **角色职责清晰，避免业务责任与系统责任混淆；**
3. **高危操作必须有人类审批，不以 AI 替代责任主体；**
4. **移动端偏简洁执行，PC 端偏复核管理；**
5. **知识沉淀必须经过审核，不把原始工单直接当正式知识。**
