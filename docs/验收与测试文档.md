## 1. 文档目的

**当前文档版本：V1.1。**

本文档在《MVP 产品需求文档》《接口文档》《数据字典与数据库设计文档》《检修业务流程与 SOP 文档》之下，给出首期 **验收标准**、**测试类型划分**、**P0 可执行用例**（含前置条件、步骤、期望结果）、**最小演示闭环（E2E）脚本**，以及与非功能需求（NFR）对齐的抽样方法，供：

- 开发自测与联调 checklist；
- QA / 团队成员编写自动化用例（pytest、Postman、k6 等）时的需求溯源；
- 答辩与交付时的 **DoD 对照清单**（与 MVP **§10** 一一挂钩）。

**检修业务 API 前缀**：`{PREFIX} = /api/v1/maintenance`（与《接口文档》一致）。下文路径均省略 `http://{host}`，写作 `{PREFIX}/...`。

### 1.1 契约与拍板来源

本文档中的 **HTTP 状态码、`business_code`、分页 `data` 形状、登出 204、附件下载 302、上传超限 413、工步/状态冲突 409、检索软失败统一 200** 等断言，**默认与《接口文档》V1.1 当前拍板（含 §3.4.1、§3.5、§3.7）一致**。若实现调整，**须先修订《接口文档》并升版，再同步修订本文对应用例与期望**，避免「测试文档与接口契约各说各话」。

## 4. 统一断言规范（与《接口文档》V1.1 对齐）

### 4.1 响应包

| 检查项 | 通过条件 |
| ------ | -------- |
| 成功 | HTTP 2xx 且 `success === true`（若实现统一响应包） |
| 业务失败（硬） | HTTP 4xx 或 5xx，`success === false`，`business_code` 与文档一致 |
| 软失败（检索降级） | HTTP **200**、`success === false`、`business_code` ∈ `EMPTY_HIT` \| `LOW_CONFIDENCE` \| `MODEL_UNAVAILABLE` |

**禁止**：同一软失败混用 200 与 4xx。

### 4.2 分页列表

凡声明为分页的 `GET` 列表，成功时 `data` **必须**为：

```json
{ "items": [], "total": 0, "page": 1, "page_size": 20 }
```

自动化断言须校验 **不存在** `records`、`list`、`pagination` 等替代键名。

### 4.3 鉴权

| 场景 | 期望 |
| ---- | ---- |
| 无 `Authorization` 访问需认证接口 | **401**，`business_code` 含 `UNAUTHORIZED`（或与实现一致且文档化） |
| Token 有效但角色不允许 | **403** `FORBIDDEN` |

## 6. 工单状态与动作白名单（验收抄表）

与《接口文档》**§8.4.1** 一致；验收时以 **服务端返回的 `work_order.status`** 为准。

| 接口 / 动作 | 允许当前工单 `status` | 成功后工单 `status`（典型） |
| ----------- | --------------------- | --------------------------- |
| `POST .../escalations` 或 `actions/request-escalation` | **S2、S3** | **S4** |
| `POST .../escalations/{id}/resolve` | **S4、S5**（且升级单允许） | **S5、S6、S7** 或退回 **S3** 等（以 OpenAPI 分支为准） |
| `POST .../actions/enter-maintenance` | **S3、S5** | **S7** |
| `POST .../approval-tasks/{id}/resolve`（`status=approved`） | **S6** 且任务匹配 | **S7** |
| `POST .../actions/complete-maintenance` | **S7** | **S8** |
| `POST .../fillings` | **S8** | **S9** |
| `POST .../steps/confirm` | 见《接口文档》§12 | 工单状态**不变**，仅推进 `current_step_no` |

非法迁移：**409** + `INVALID_STATE_TRANSITION`。

### 6.1 数据库断言（P0，与接口成功响应交叉验证）

以下在 **接口断言通过后**，对 **同一事务或紧随其后的只读查询** 执行 SQL / ORM 断言（表名、字段名以《数据字典》V1.1 为准）。

| 用例 ID | 触发操作 | 数据库期望 |
| ------- | -------- | ---------- |
| TC-DB-001 | `POST .../retrieval` 成功或软失败但返回 `retrieval_snapshot_id` | `retrieval_snapshots` 存在对应行：`work_order_id` 正确；若有 `message_id` 列或与消息表关联，则与本次返回的 `message_id` **一致**；引用片段与响应 `citations` **可对应**（冻结语义） |
| TC-DB-002 | `POST .../fillings` 成功且此前该工单已有回填 | 新行 `is_latest=true`；**所有**旧行 `is_latest=false`；仅 **一行** `is_latest=true`（按 `work_order_id`） |
| TC-DB-003 | `POST .../approval-tasks/{id}/resolve` 首次终态 | `approval_tasks` 该 `id` 为终态；**无第二条**同任务重复终态记录 |
| TC-DB-003b | 同上任务，重复相同结论（幂等） | 终态记录条数 **不增加**；审计表（若有）可有一条幂等访问记录 |
| TC-DB-004 | `POST .../escalations` 成功 | 同工单下「进行中」升级单条数 **≤1**；唯一约束或应用层锁与 **409 ESCALATION_IN_PROGRESS** 一致 |
| TC-DB-005 | `POST .../knowledge-articles/{id}/publish` 成功 | 该 `series_id` 下 **`status=published` 行数 = 1**（数据字典版本规则） |

**说明**：自动化可在 API 测试后使用测试库只读账号查询；或依赖实现暴露的 `GET` 详情二次确认（较弱，答辩时优先展示 DB 断言）。

## 8. 权限抽样矩阵（P0）

### 8.1 写接口与角色（接口级）

| 接口（方法） | worker | expert | safety | admin |
| ------------ | ------ | ------ | ------ | ----- |
| `POST .../work-orders` | ✓ | 依矩阵 | ✗ | 依矩阵 |
| `POST .../escalations/{id}/resolve` | ✗ | ✓（本人指派） | ✗ | ✗ |
| `POST .../approval-tasks/{id}/resolve` | ✗ | ✗ | ✓ | ✗ |
| `POST .../knowledge-articles/{id}/publish` | ✗ | ✗ | ✗ | ✓ |
| `GET .../admin/audit-logs` | ✗ | ✗ | ✗ | ✓ |

「✓」表示允许；「✗」期望 **403**。具体以《用户角色与使用场景文档》为准，若有偏差须在测试报告登记为 **矩阵修订** 并回写该文档。

## 10. E2E-01：最小演示闭环脚本（MVP §2.4）

**目的**：答辩录屏一条过；步骤可映射到页面（MVP §5.1）。

| 步骤 | 操作者 | 动作（API 或 UI） | 期望状态 / 数据 |
| ---- | ------ | ----------------- | ---------------- |
| 1 | admin | 确保设备 A 已配置责任专家；知识库已导入可命中片段 | 环境就绪 |
| 2 | worker | `login` → `GET devices` 选择设备 A | 200 |
| 3 | worker | `POST attachments` 上传 ≥1 张图 | 得 `attachment_ids` |
| 4 | worker | `POST work-orders` → 得 `work_order_id` | `S1` |
| 5 | worker | `POST .../retrieval`（带 `query_text` 与附件） | 成功或软失败均为 **200**；成功则 `citations` 非空优先 |
| 6 | worker | `GET .../messages` 核对 assistant 与快照 | 与 §8.0 一致 |
| 7 | worker | **与《接口文档》V1.1 对齐的最小 API 路径（无高危分支演示）**：在工单已达 **S3**（通常由检索成功写入）后，依次 `POST .../actions/enter-maintenance`（**S3→S7**）→ `POST .../actions/complete-maintenance`（**S7→S8**）。若演示含工步引导，在 **S7** 插入 `GET flow-templates`、`POST .../steps/confirm`（须满足 §12 前置） | 每步 **200**，`GET work-orders/{id}` 的 `status` 与 §6 表一致 |
| 8 | worker | `POST .../fillings`：`resolved` + 合法 `closure_code` + `attachment_ids`≥1 | 工单 **S9**；**§6.1 TC-DB-002** 可执行 |
| 9 | expert / admin | **复核结单（S9→S10）**：《接口文档》V1.1 **未单列**该语义化动作时，首版 E2E 允许：(a) 在《测试报告》声明经 **管理端/受控脚本** 将工单置 **S10**；或 (b) 实现补充 `POST .../work-orders/{id}/actions/...` 后在 OpenAPI 与本文同步更新。**答辩前须三处一致**：接口、本脚本、测试报告 | **S10** 或报告声明的等价已结单 |
| 10 | expert/admin | `from-work-order` 或等价生成候选 → `review` approve → admin `publish` | 知识 **published** |
| 11 | worker | 再次 `POST .../retrieval` 相同或相近 query | 命中新发布内容（FR-KB-03） |

**说明**：步骤 7 以 **§6 状态白名单** 为判据；步骤 9 的缺口须在报告 **「E2E 裁剪声明」** 中单条写明，并与 MVP DoD、接口版本号对照。

## 12. NFR 与性能抽样（与 MVP §7 对齐）

### 12.1 方法学前提（须在《测试报告》「环境与方法」中披露）

| 项 | 建议写法 |
| -- | -------- |
| **冷启动** | 首条检索请求可作为 **warm-up 不计入** P95；或单独记录「首次冷启动延迟」与「稳态 P95」两列 |
| **并发模型** | **首条建议 P95**：默认 **串行** 重复 N 次（如 N=20～30），**非** k6 全压并发，除非另开「容量测试」章节 |
| **环境** | 记录：CPU 型号/核数、内存、磁盘类型；服务端与 DB **同机或分离**；是否 Docker |
| **知识库规模** | 固定 **chunk 条数 / 索引体积**（或注明「演示库全量」），版本变更须重记 |
| **云端模型** | 对外部 LLM **不可控延迟**：须记录 **测试时段、地域、模型名**；若偶发超时，记为 **环境事件** 而非直接判接口失败（与 FR-RAG-06 一致） |
| **时钟** | 使用 NTP 同步；记录客户端与服务端时区 |

### 12.2 指标与执行

| 指标 | 方法 | 记录项 |
| ---- | ---- | ------ |
| 首条建议 P95 ≤ 3s | 按 **§12.1** 串行 N 次，取 P95；排除约定 warm-up | 环境、KB 规模、模型端点 |
| 上传 10MB 边界 | TC-ATT-002 | 实际返回 **413**、`PAYLOAD_TOO_LARGE` |
| 审批写库 P95 ≤ 500ms | 串行仅测 `approval-tasks/.../resolve` 写路径 | 单机/分离部署声明 |

结果写入《测试报告》**环境与方法**章节；未达标记为 **已知风险** 并关联 issue。
